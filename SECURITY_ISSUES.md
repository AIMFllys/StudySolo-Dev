# Security Audit Report

This document outlines the potential security vulnerabilities identified during the codebase audit.

## 1. Email Verification Code Prediction (Insecure Randomness)
- **Location**: `backend/app/services/email_service.py` -> `_generate_code()`
- **Description**: The email verification code is generated using `random.choices()`, which is based on the Mersenne Twister pseudo-random number generator. This PRNG is not cryptographically secure, and its output can be predicted if an attacker observes enough outputs, potentially allowing them to bypass the email verification mechanism.
- **Recommendation**: Replace `random.choices()` with `secrets.choice()` or `secrets.randbelow()`, which use a cryptographically secure random number generator (CSPRNG) suitable for generating secure tokens and codes.

## 2. IP Spoofing for Rate Limit Bypass
- **Location**: `backend/app/api/auth/_helpers.py` -> `resolve_client_ip()`
- **Description**: The function blindly trusts the `X-Forwarded-For` HTTP header to determine the client's IP address. Since this header can be easily spoofed by the client, an attacker can bypass IP-based rate limiting (e.g., in `/register` or `/reset-password-with-code`) by constantly changing the `X-Forwarded-For` value.
- **Recommendation**: Only trust `X-Forwarded-For` or similar headers if they are explicitly appended by a trusted reverse proxy (like Nginx, Cloudflare). Configure the application to only accept these headers from known proxy IP addresses.

## 3. Race Condition / TOCTOU in Captcha Verification
- **Location**: `backend/app/api/auth/captcha.py` -> `consume_captcha_token()`
- **Description**: The logic to verify and consume a captcha token suffers from a Time-Of-Check to Time-Of-Use (TOCTOU) race condition. It first queries the database to check if the token is verified and not consumed, then separately updates the database to mark it as consumed. A concurrent attacker could send multiple requests simultaneously to use the same token multiple times before the first request successfully marks it as consumed.
- **Recommendation**: Perform an atomic check-and-update operation. For example, use a single `UPDATE` query that filters on `consumed = False` and updates it to `True`, and then verify if any rows were affected.

## 4. Insecure File Download / IDOR
- **Location**: `backend/app/api/exports.py` -> `download_export()`
- **Description**: The endpoint allows logged-in users to download files via their filenames. While it correctly checks for path traversal, it lacks ownership checks. This is an Insecure Direct Object Reference (IDOR) vulnerability, meaning any authenticated user can guess or brute-force filenames (even if they are UUID-based) and download exports generated by other users.
- **Recommendation**: The database or the server needs to keep track of the file ownership, or the files should be stored inside user-specific directories (e.g., `EXPORT_DIR/{user_id}/{filename}`). The endpoint must verify that the requesting `current_user` actually owns the file before serving it.
