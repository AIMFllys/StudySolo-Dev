# StudySolo Backend Security Vulnerability Report

## 1. Captcha Mechanism Vulnerable to Rainbow Table / Pre-computation Attack

**Location**: `backend/app/api/auth/captcha.py`

**Vulnerability Description**:
The slider puzzle captcha challenge generates a random seed to compute the target position `target_x`. The random seed is generated as follows:
```python
seed = int.from_bytes(os.urandom(4), "big") % 100000
```
Because the seed is modulo 100,000, there are only 100,000 possible seeds. The `target_x` calculation:
```python
def _compute_target_x(seed: int) -> int:
    rng = _mulberry32(seed)
    ...
```
is completely deterministic and strictly dependent on this seed.

When generating a challenge, the server returns the seed to the frontend:
```python
return {"seed": seed, "challenge": challenge}
```
An attacker can easily pre-compute all 100,000 possible `target_x` values for each possible seed and store them (e.g., a rainbow table). Because 100,000 is a very small number, this pre-computation takes less than a second on any modern CPU. The attacker can then script a bypass for the captcha with a 100% success rate by looking up the correct `target_x` for the provided `seed` in their table.

**Impact**:
High. The captcha mechanism can be trivially bypassed, allowing attackers to abuse endpoints protected by it, such as email registration code sending (`/api/auth/send-code`), opening the door to SMS/Email bombing and automated mass registration.

**Remediation**:
Do not expose the raw seed or predictable parameters that dictate the answer to the client. The challenge should be completely server-side generated and only the puzzle image/pieces should be sent to the frontend. The valid answer (`target_x`) should be stored in the database associated with an unpredictable `challenge_id` without giving away the mathematical derivation seed to the client.

## 2. Path Traversal and Inconsistent Temp Directory Logic in File Exports

**Location**: `backend/app/api/exports.py` and `backend/app/services/file_converter.py`

**Vulnerability Description**:
In `app/api/exports.py`, the download endpoint uses the following logic to resolve the export directory:
```python
EXPORT_DIR = os.getenv("EXPORT_DIR", os.path.join(os.path.expandvars("%TEMP%"), "studysolo_exports"))
```
Whereas in `app/services/file_converter.py`, it resolves it as:
```python
EXPORT_DIR = os.getenv("EXPORT_DIR", os.path.join(tempfile.gettempdir(), "studysolo_exports"))
```
On Linux or macOS environments, `%TEMP%` is not automatically expanded unless there is explicitly an environment variable named exactly `%TEMP%`. `os.path.expandvars` will just return the literal string `"%TEMP%"`. This will result in the `exports.py` endpoint attempting to access a directory named `%TEMP%/studysolo_exports` relative to the current working directory, which will fail to find files generated by `file_converter.py` (which correctly uses the system's temp dir via `tempfile.gettempdir()`).

Furthermore, in `exports.py`, the path traversal check is flawed:
```python
    real_path = os.path.realpath(filepath)
    real_export = os.path.realpath(EXPORT_DIR)
    if not real_path.startswith(real_export):
        raise HTTPException(status_code=403, detail="访问被拒绝")
```
Using `.startswith()` on strings for path checking is a well-known anti-pattern. If `real_export` resolves to `/tmp/studysolo_exports` (without a trailing slash), an attacker could potentially access `/tmp/studysolo_exports_malicious/...` if such a directory happened to exist, because `/tmp/studysolo_exports_malicious` starts with `/tmp/studysolo_exports`.

**Impact**:
Moderate. File downloads will constantly fail (404) on Unix-like environments if `EXPORT_DIR` is not explicitly set via environment variables. The path traversal vulnerability is limited but represents poor security hygiene.

**Remediation**:
1. Unify the `EXPORT_DIR` configuration in a single core config file (e.g., `app/core/config.py`) and use `tempfile.gettempdir()` uniformly. Do not use `os.path.expandvars("%TEMP%")`.
2. Fix the path validation logic using `os.path.commonpath`:
```python
if os.path.commonpath([real_export, real_path]) != real_export:
    raise HTTPException(...)
```
