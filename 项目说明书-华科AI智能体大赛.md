## 一、问题背景

当前，以 ChatGPT、DeepSeek 为代表的大语言模型已经深入学生日常学习。然而，真实学习场景中存在一个普遍而尚未解决的核心矛盾：

**单轮对话无法覆盖完整的学习过程。**

一个典型的学习任务——例如"系统性掌握机器学习基础"——需要经历"目标拆解 → 大纲生成 → 内容提取 → 知识总结 → 闪卡记忆 → 测验评估"等多阶段流程。目前的解决方式存在以下问题：

| 现有方案 | 局限性 |
|---------|-------|
| 通用聊天机器人 | 只能逐轮问答，无法编排多步骤学习流程；前一步产出无法自动流入后一步 |
| 传统工作流平台（如 Dify、Coze） | 面向通用场景，缺乏学习领域专用节点；工作流搭建门槛高，需要理解 DAG 概念和 JSON 配置 |
| 手动拼接多个 AI 工具 | 复制粘贴成本高；无法复用和共享流程；学习过程不可观测 |

我们认为，下一代学习辅助工具的核心能力不应仅停留在"生成一段文字"，而应当提供：
- 让 AI 理解学习目标并**自动规划**步骤
- 让各步骤以**可视化工作流**形式编排，流程可回溯、可修改
- 让学习流程可**保存、共享、复用**，形成社区化知识资产

**StudySolo 正是为解决以上问题而设计的面向学习场景的智能体可视化编排平台。**

## 二、需求分析

### 2.1 目标用户

本平台面向大学生及终身学习者，覆盖"从零构建学习计划"到"系统性复习巩固"的全流程。

### 2.2 核心需求

**需求一：自然语言驱动的工作流生成**

用户不需要学习工作流搭建的专业知识，只需在侧边栏用自然语言描述学习目标，例如"帮我做一个关于线性代数的学习计划"，系统便自动生成包含多个步骤的可视化工作流。

**需求二：可视化编辑与灵活调整**

生成的工作流以画布形式展示，用户可以拖动节点、调整连接关系、添加或删除步骤。这种"所见即所得"的交互方式降低了使用门槛，同时保留了高级用户对流程的精细控制能力。

**需求三：流式执行与全程可观测**

点击执行后，各节点按依赖顺序逐步运行，每个节点的输入、输出、耗时、状态均实时可见。用户能清晰地看到"AI 在每一步做了什么"，这是传统聊天机器人无法提供的透明度。

**需求四：工作流共享与社区协作**

好的学习流程不应局限于个人使用。用户可将自己的工作流公开发布，其他用户可以浏览、收藏、分叉并在此基础上改进，形成学习社区的协作生态。

**需求五：节点共建与持续扩展**

平台预置了 18 种学习专用节点，同时支持用户自行创建并发布提示词节点。未来还将支持 MCP 协议接入和外部 API 对接，使平台的能力边界持续扩展。

## 三、智能体架构

### 3.1 核心设计理念：节点即智能体，工作流即多智能体协作

StudySolo 的智能体架构建立在一个核心抽象之上：**每个工作流节点本身就是一个独立的智能体**。

每个节点具备：
- **独立的 System Prompt**：定义该智能体的角色与行为约束
- **独立的模型选择**：可针对任务特性选择最适合的大模型
- **独立的输入/输出契约**：通过 ExecutionContext 与其他智能体交换数据
- **独立的配置能力**：通过 config_schema 声明可调参数

当多个节点通过有向无环图（DAG）连接时，本质上构成了一个**多智能体协作系统**：上游智能体的输出作为下游智能体的输入，由 DAG 执行引擎按拓扑序协调执行。

### 3.2 四层智能体架构

```
Layer 0 ─ 意图理解层（Intent Classifier）
  │  用户在侧边栏输入自然语言 → AI 分类为 BUILD / MODIFY / CHAT / ACTION
  ▼
Layer 1 ─ 规划层（AI Planner Agent）
  │  将学习目标分解为多步骤工作流图（含节点、连线、初始配置）
  ▼
Layer 2 ─ 编排层（DAG Orchestrator）
  │  基于拓扑排序确定执行顺序，处理分支/合并/循环逻辑
  ▼
Layer 3 ─ 执行层（18 种 Node Agent）
  │  每个节点独立执行，通过 SSE 实时回传进度与结果
  ▼
Layer 4 ─ 路由层（AI Model Router）
     8 个 AI 平台 × 3 种路由策略，自动选择最优模型并支持多级降级
```

### 3.3 意图理解与画布交互

系统的侧边栏 AI 面板支持三种交互模式：

| 模式 | 功能 | 输出 |
|------|------|------|
| 对话模式 | AI 导师角色，回答学习问题 | Markdown 文本流 |
| 规划模式 | 分析当前工作流并给出改进建议 | XML 结构化建议（含优先级） |
| 创建模式 | 直接操作画布——自动新增/删除/修改节点和连线 | JSON 操作指令 |

**创建模式的核心创新**在于：AI 能够**"看见"当前画布**。前端在每次请求时将画布上所有节点的类型、标签、状态、上下游关系及坐标序列化为结构化文本，注入到大模型的 System Prompt 中。模型返回的操作指令由 ActionExecutor 解释执行，并支持快照式撤销回滚，确保操作安全。

### 3.4 节点体系（18 种学习专用智能体）

| 类别 | 节点 | 功能 |
|------|------|------|
| **输入类** | trigger_input | 工作流入口，承载用户原始学习目标 |
| | knowledge_base | 知识库节点，支持文件上传与文档内容注入 |
| | web_search | 联网搜索节点，支持多源并发检索 |
| **分析类** | ai_analyzer | 深度分析，对输入内容进行多维度拆解 |
| | ai_planner | 智能规划，将学习目标分解为结构化方案 |
| | content_extract | 内容提取，从长文本中抽取关键信息 |
| | compare | 对比分析，多角度比较不同概念或方案 |
| **生成类** | outline_gen | 大纲生成 |
| | summary | 总结归纳 |
| | flashcard | 闪卡生成（JSON 结构化输出） |
| | quiz_gen | 测验生成（JSON 结构化输出） |
| | mind_map | 思维导图生成 |
| | merge_polish | 多源合并与润色 |
| **交互类** | chat_response | 对话回复 |

### 3.5 智能模型路由

后端 AI Router 对接了 8 个国内主流大模型平台，实现统一调度：

- **DeepSeek**（V3 / R1）
- **通义千问**（Turbo / Plus / Max / Qwen3-Max）
- **智谱 GLM**（Flash / GLM-4 / GLM-4.5 / GLM-4.7）
- **豆包**（Pro / Pro-256K / Seed 2.0）
- **月之暗面 Kimi**（8K / 128K / K2 / K2.5）
- **七牛云**（代理通道）
- **硅基流动**（代理通道）
- **火山引擎**（原生接入）

支持 3 种路由策略：`native_first`（原生优先）、`proxy_first`（代理优先）、`capability_fixed`（能力固定）。每个节点类型配置有 2-3 个 SKU 候选，单个平台宕机时自动切换至备用通道，实现多级容灾降级。

## 四、技术方案

### 4.1 整体技术架构

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | Next.js 15 + React 19 + TypeScript + Tailwind CSS v4 | 基于 App Router 的现代 SSR 架构 |
| 画布引擎 | @xyflow/react（React Flow） | 支持节点拖拽、连线、分组、缩放的工业级画布 |
| 状态管理 | Zustand | 轻量级，支持 Undo/Redo 快照 |
| 后端 | Python + FastAPI | 异步框架，原生支持 SSE 流式推送 |
| 执行引擎 | 自研 DAG Executor | 基于拓扑排序 + ExecutionContext 黑板模型 |
| 数据库 | Supabase (PostgreSQL) + Row Level Security | 多租户安全隔离 |
| 认证 | Supabase Auth + JWT + 自定义验证码 | Canvas 拼图验证码 + IP 锁定 |
| 部署 | 阿里云 ECS + Nginx + Baota 面板 | 已完成生产环境部署 |

### 4.2 DAG 执行引擎（核心）

执行引擎（executor.py）是平台的核心组件，负责将用户构建的工作流图转化为可执行的任务序列：

**执行流程**：
1. 从前端 POST 请求体中获取当前画布的 nodes 与 edges（确保执行的是用户眼前的最新图，而非数据库中的旧版本）
2. 对 DAG 进行拓扑排序，确定执行顺序
3. 逐节点执行，通过 ExecutionContext 在节点间传递中间结果
4. 每个节点执行过程通过 SSE 实时推送 7 种事件：`node_input`、`node_status`、`node_token`、`node_done`、`loop_iteration`、`save_error`、`workflow_done`

**前端实时追踪**：
- 执行面板以时间线形式展示每个节点的运行状态（等待/运行/完成/错误）
- 支持按 root-to-leaf 链路筛选查看，分支工作流可按线路查看
- 流式 Token 实时追加，实现打字机效果

### 4.3 数据同步机制

工作流数据采用双缓冲同步策略：
- **本地缓冲**：IndexedDB 快照，300ms 防抖，确保离线可恢复
- **云端同步**：Supabase 数据库，5s 节流，确保多端一致性
- 执行时直接提交当前内存中的图结构，从根本上解决"编辑态与执行态不一致"的问题

### 4.4 Prompt 模块化系统

所有 AI 提示词采用 Markdown 文件 + 模板变量的架构，而非硬编码在代码中：

```
backend/app/prompts/
├── identity.md            ← 全局身份设定（所有模式共享）
├── intent_classifier.md   ← 意图分类器
├── mode_chat.md           ← 对话模式提示词
├── mode_plan.md           ← 规划模式提示词
├── mode_create.md         ← 创建模式提示词
└── nodes/
    └── _base_prompt.md    ← 节点通用约束

backend/app/nodes/<category>/<node_type>/
└── prompt.md              ← 每个节点独立的 System Prompt
```

提示词通过 `prompt_loader.py` 统一加载，支持 `{{变量}}` 模板渲染和 LRU 缓存。这使得提示词可独立于代码迭代，支持 Git 版本管理和 A/B 测试。

## 五、创新点

### 创新一：自然语言驱动的工作流生成与操作

用户无需学习工作流搭建知识，通过侧边栏自然语言对话即可完成：
- **从零构建**：描述学习目标 → AI 自动生成完整工作流图（含节点、连线、初始配置）
- **增量修改**：说"在总结后面加一个闪卡节点" → AI 直接操作画布完成添加
- **智能分析**：切换规划模式 → AI 审视当前工作流结构并给出改进建议

底层通过画布序列化 + 意图分类 + 指令执行器实现，AI 能够"看见"并"操作"画布。

### 创新二：社区化工作流共享与分叉

工作流不是一次性产物，而是可复用的学习资产：
- 用户可将优秀工作流公开发布至社区
- 其他用户可浏览、收藏、点赞
- 支持分叉（Fork）——在他人工作流基础上修改，形成自己的版本
- 积累社区集体智慧，降低学习流程设计的门槛

### 创新三：开放式节点共建生态

平台不仅提供 18 种预置节点，还支持用户自行创建和发布提示词节点：
- 用户定义节点的 Prompt、输入提示、输出格式
- 支持上传知识文件作为节点的领域知识注入
- JSON 输出格式支持 AI 辅助生成 Schema 约束
- 发布后进入社区节点商店，其他用户可直接拖入自己的工作流使用
- 封装后的 Prompt 对使用者不可见，保护创作者的知识产权

这种机制使得平台能力不局限于开发团队，而是由整个用户社区共同扩展——**用户既是使用者，也是构建者**。

### 创新四：完整的 SOP 化开发流程

我们将"新增节点"这一关键开发动作标准化为完整的 SOP 文档体系。所有规范均已在 GitHub 仓库中开源：https://github.com/AIMFllys/StudySolo

- **A型-LLM提示词节点-SOP.md**：`docs/项目规范与框架流程/功能流程/新增AI工具/A型-LLM提示词节点-SOP.md`
- **B型-外部工具节点-SOP.md**：`docs/项目规范与框架流程/功能流程/新增AI工具/B型-外部工具节点-SOP.md`
- **01-现有节点功能补全总则.md**：`docs/项目规范与框架流程/功能流程/新增AI工具/01-现有节点功能补全总则.md`

这套 SOP 已封装为开发技能（Skills），支持 AI 辅助开发时自动参考，显著降低后续节点扩展的开发成本。新增一个 A 型节点，后端不超过 3 个文件，前端不超过 4 个文件。

### 创新五：面向生产的平台治理能力

区别于纯 Demo 级项目，StudySolo 具备完整的生产级运营能力：
- **管理后台**：10 个管理模块，覆盖用户管理、模型配置、SKU 管理、使用量统计、审计日志等
- **计费体系**：基于 SKU 的模型调用成本核算，支持会员分层（Free / Pro / Pro+ / Ultra）
- **安全机制**：Canvas 拼图验证码、IP 锁定、Row Level Security、CAPTCHA 防暴力破解

## 六、应用场景

### 场景一：系统化学习一个新知识领域

用户输入"帮我制定机器学习入门学习计划"，系统自动生成包含"目标拆解 → 大纲生成 → 内容提取 → 总结归纳 → 闪卡生成 → 测验评估"的完整工作流。执行后得到结构化的学习材料，可导出为文件或写入数据库留存。

### 场景二：课后复习与知识巩固

用户上传课程 PDF 至知识库节点，接入"内容提取 → 闪卡生成"工作流，快速将课件内容转化为记忆卡片。高频使用的复习流程可保存为模板，下次直接复用。

### 场景三：社区共享优秀学习流程

一位擅长论文阅读的用户构建了"论文精读流程"（摘要提取 → 方法论分析 → 创新点提炼 → 批判性评价），发布至社区。其他用户直接使用或分叉修改，不需要从零设计流程。

### 场景四：教师辅助教学设计

教师构建"知识点讲解 → 随堂测验 → 错题分析 → 补充学习"工作流，部署后供学生自主学习。教师可通过管理后台查看使用统计。
