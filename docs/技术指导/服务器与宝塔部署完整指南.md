# 🚀 1037Solo 小白级服务器与宝塔全自动部署指南

> **文档状态**：🟢 正式发布 (2026-03-26)
> **适用对象**：零基础开发者、需要了解 GitHub + 宝塔面板 (BT Panel) CI/CD 自动化部署的团队成员。
> **核心依赖**：遵循本项目的 `deployment-procedures` 与生产环境规范。

## 🧭 1. 部署核心原则（必读）

根据项目的 `deployment-procedures` 技能规范，任何部署都必须遵循 **5阶段部署流程 (5-Phase Process)**：

1. **PREPARE (准备)**：确保代码本地测试通过、环境变量无误、生产环境 Build 成功。永远不要部署未经测试的代码。
2. **BACKUP (备份)**：宝塔面板支持一键备份目录和数据库。没有备份 = 没有退路。
3. **DEPLOY (部署)**：执行部署时，必须全程盯着屏幕看日志，切忌“点完不管”。
4. **VERIFY (验证)**：部署后立即检查应用健康状态（前台能否打开、API 是否联通）、查看错误日志。
5. **CONFIRM or ROLLBACK (确认/回滚)**：如果出现严重错误、白屏、500/502，**立即回滚（Rollback）**，不要在生产环境查Bug！

**安全部署红线**：
- ❌ 不要周五下午发布。
- ❌ 不要在一次部署中混杂多项改动（One change at a time）。
- ✅ 重视监控：前 5 分钟死盯日志，15 分钟确认稳定。

## 🏗️ 2. 项目架构与端口规划全景

在宝塔中部署之前，你必须知道我们的服务是如何分布的。1037Solo 是一个**混合架构 (Hybrid Architecture)**，由 Node.js 和 Python 共同组成。

所有外部请求统一进入 **Nginx 80/443端口**，再由 Nginx 分发给内部微服务：

| 服务名称 | 框架类型 | 内部端口 | 宝塔管理方式 | 对应域名路径 |
| :--- | :--- | :--- | :--- | :--- |
| **Solo-API** | Node.js (Express) | `1036` | PM2 | `main.api.1037solo.com` |
| **StudySolo 前端** | Node.js (Next.js) | `2037` | PM2 | `studyflow.1037solo.com` |
| **StudySolo 后端** | Python (FastAPI) | `2038` | CLI (Supervisor/Gunicorn) | `studyflow.1037solo.com/api/` |
| **Chat 前端** | Node.js (Next.js) | `3036` | PM2 | `platform.1037solo.com/chat/` |
| **Chat 后端** | Python (FastAPI) | `3039` | CLI (Uvicorn) | Nginx 内部反代使用 |

> **⚠️ 核心防坑指南**：**外网映射 (External Mapping)** 绝对不能开启！所有微服务必须绑定在 `127.0.0.1`，仅通过 Nginx 反向代理向外暴露，否则你的内部接口会被外网直接看光。

## 🛠️ 3. 宝塔面板 (BT) 手把手建站

### 步骤 A：环境基础准备
1. 登录宝塔面板，进入「软件商店」。
2. 安装并启动 **Nginx** (用于网关)。
3. 安装 **PM2管理器 / Node.js版本管理器**，安装 Node.js `v18.x` 或 `v20.x`版本。
4. 安装 **Python项目管理器**，安装 Python `3.11+` 环境（**绝对不能用默认的3.6版**）。

### 步骤 B：创建 Node.js 项目 (以前端为例子)
1. 进入宝塔的 **网站 (Websites)** -> **Node项目** -> **添加 Node 项目**。
2. **项目目录**：选择你代码所在的目录，例如 `/www/wwwroot/studyflow.1037solo.com/frontend`。
3. **启动选项**：
   - **启动命令**：填写 `start` (相当于 `npm run start`)
   - **项目端口**：填写 `2037`
   - **运行参数**：填写 `-p 2037` (强制绑定)
   - **绑定域名**：**留空！留空！留空！**（由 Nginx 统一管理）
   - **开启 PM2**：打勾 ✅

### 步骤 C：创建 Python 项目 (以后端为例)
1. 进入宝塔 **网站** -> **Python项目** -> **添加 Python 项目**。
2. **项目名称**：`studysolo-backend`
3. **启动方式**：选择 **Command Line (命令行启动)**。
4. **项目目录**：`/www/wwwroot/studyflow.1037solo.com/backend`
5. **启动命令**：`uvicorn app.main:app --host 127.0.0.1 --port 2038 --workers 4`
6. **运行用户**：`www`
7. 注意：**外网映射千万不要打勾！**

## 🚪 4. Nginx 统一网关与路径拦截 (重点)

千万不要给每个 Node/Python 建独立的域名站点，我们采用的是 **Unified Gateway (统一网关)**。

1. 在宝塔的「**PHP项目** / **静态网站**」里创建一个站点，域名填 `studyflow.1037solo.com`。
2. 点击网站设置 -> **配置文件**，添加我们专属的反向代理代码：

```nginx
server {
    listen 80;
    listen 443 ssl http2;
    server_name studyflow.1037solo.com;
    
    # SSL 证书配置略...

    # =============================================
    # 微服务路由分发逻辑
    # =============================================
    
    # 1. 拦截前端路由 (转给PM2 Node服务)
    # ⚠️ 必须带 ^~，防止宝塔自带的静态正则匹配覆盖掉 next.js 的请求
    location ^~ / {
        proxy_pass http://127.0.0.1:2037;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # 2. 拦截后端 API (转给 FastAPI)
    location ^~ /api/ {
        proxy_pass http://127.0.0.1:2038;
        
        # ⚠️ AI Streaming 优化 (极其重要) 
        # 如果没有下面这三句，打字机流式效果(SSE)会卡着出不来！
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 300s;
    }
}
```

> **致命坑点（Trailing Slash 循环重定向）**：
> 如果 Nginx 和 Next.js 在 URL 结尾斜杠 `/` 上意见不一，会导致无限 308 重定向死循环。
> 解决办法：在代码 `next.config.ts` 中强制加上 `trailingSlash: true` 配置。

## 🤖 5. GitHub Workflows: 一键自动化部署

平时改完代码还要 FTP 上传？还要去服务器手动敲 PM2 重启？太痛苦了！
这里教你如何纯靠 `git push`，全自动完成“上传代码 -> 安装依赖 -> 打包构建 -> 重启宝塔服务”。

### 第一步：在 GitHub 配置安全密钥 (Secrets)
进入 GitHub 项目页面 -> **Settings** -> **Secrets and variables** -> **Actions**。添加以下三个 **Repository secrets**：
- `SERVER_IP`：你的宝塔服务器外网 IP。
- `SERVER_USER`：服务器用户名（通常是 `root`）。
- `SERVER_SSH_KEY`：服务器的 SSH 私钥（`~/.ssh/id_rsa` 的内容。没有的话，用 `ssh-keygen` 在服务器生成，然后把 `.pub` 添加到 `~/.ssh/authorized_keys` 中）。

### 第二步：创建 Workflow 文件
在项目根目录，新建文件 `.github/workflows/deploy.yml`。

内容如下，这是一个标准的**远程 SSH 执行规范**模板：

```yaml
name: Deploy StudySolo to BT Panel

on:
  push:
    branches:
      - main  # 当你推送代码到 main 分支时自动触发

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: 🚀 1. Checkout Code
        uses: actions/checkout@v3

      # 这里我们采用“在服务器直接执行脚手架命令”的方式。
      # 如果你的服务器内存小于 2G，推荐先在 local 打包（build）再通过 scp-action 上传。
      - name: 🔧 2. Execute Deployment via SSH
        uses: appleboy/ssh-action@v0.1.7
        with:
          host: ${{ secrets.SERVER_IP }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          script: |
            # 1. 切换到宝塔网站根目录
            cd /www/wwwroot/studyflow.1037solo.com
            
            # 2. 拉取最新代码 (这要求你的服务器之前已经 clone 过这个项目)
            echo "📥 Pulling latest code..."
            git reset --hard HEAD
            git pull origin main
            
            # 3. 部署前端 (Next.js)
            echo "📦 Building Frontend..."
            cd frontend
            # 安装依赖（推荐用 pnpm）
            pnpm install
            # 生产环境打包
            pnpm run build
            # 重启宝塔中的 PM2 服务 (2037 端口进程)
            pm2 reload 1037solo-frontend
            cd ..
            
            # 4. 部署后端 (Python FastAPI)
            echo "🐍 Updating Backend..."
            cd backend
            # 激活宝塔 python 项目所在虚拟环境 (例如 /www/.../backend/xxxx_venv/bin/activate)
            # 或者让宝塔 Python 项目管理器自动识别重载
            source /www/wwwroot/studyflow.1037solo.com/backend/[改成了你的虚拟环境名_venv]/bin/activate
            # 更新包
            pip install -r requirements.txt
            # 如果配置过 systemd 会直接 restart，这里以重启 supervisor / kill uvicorn 为例
            pkill -f "uvicorn app.main:app" || true
            # (宝塔如果使用了守护进程，kill 掉后会自动唤醒，这里仅作无缝平滑演示)
            
            echo "✨ Deploy completed successfully!"
```

### 第三步：如何工作？
1. 本地修改完代码，跑完测试。
2. 控制台输入 `git add .` -> `git commit -m "feat: new page"` -> `git push`。
3. 喝杯咖啡，打开 GitHub 的 **Actions** 标签卡，你能实时看到云端有个机器人在帮你输入命令、打包、并按下了重启按钮。
4. **验证（VERIFY）**：等 Action 亮起绿灯✅后，立刻访问网站，确认一切正常。如果不正常，本地立刻 `git revert` 并再次 push！这就是自动化部署的终极兜底艺术。

> **小结总结**：真正的后端开发，不仅是代码能跑，还要考虑**高可用、流式无阻断、零冲突反代和一键发布**。跟着这篇指南，你就拥有了企业级的部署思维。
