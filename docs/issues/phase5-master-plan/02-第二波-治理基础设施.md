## 1. S2.1 前端裸 fetch 统一收口

### 1.1 问题描述

文档 `02-模块边界规范.md` 明确规定所有 API 调用必须通过 `services/` 层的 `authedFetch`，但实际存在 6 处裸 `fetch` 绕过认证链：

| 文件 | 行号 | 调用目标 | 风险 |
|------|------|---------|------|
| `features/workflow/components/panel/WorkflowPromptInput.tsx` | 39 | `/api/ai/generate-workflow` | 缺少 token 刷新重试 |
| `features/workflow/hooks/use-create-workflow-action.ts` | 25 | `/api/workflow` | 缺少统一错误处理 |
| `features/workflow/hooks/use-stream-chat.ts` | 132 | `/api/ai/generate-workflow` | 缺少 token 刷新重试 |
| `features/workflow/hooks/use-stream-chat.ts` | 251 | `/api/ai/chat-stream` | SSE 流式，需特殊处理 |
| `features/workflow/hooks/use-workflow-execution.ts` | 106 | `/api/workflow/{id}/execute` | SSE 流式，需特殊处理 |
| `features/workflow/hooks/use-workflow-sync.ts` | 185 | `/api/workflow/{id}` | 缺少统一错误处理 |

### 1.2 修复策略

分两类处理：

**普通 JSON 请求（4 处）**：直接替换为 `authedFetch`
- `WorkflowPromptInput.tsx:39`
- `use-create-workflow-action.ts:25`
- `use-stream-chat.ts:132`
- `use-workflow-sync.ts:185`

**SSE 流式请求（2 处）**：需要在 `api-client.ts` 中新增 `authedStreamFetch` 或确认现有 `authedFetch` 是否支持 SSE
- `use-stream-chat.ts:251`
- `use-workflow-execution.ts:106`

### 1.3 操作步骤

1. 检查 `services/api-client.ts` 中 `authedFetch` 的实现，确认是否支持 SSE 场景
2. 如不支持，新增 `authedStreamFetch`：复用认证逻辑，但返回 `Response` 而非解析后的 JSON
3. 逐文件替换裸 `fetch`，每次替换后手动 smoke 验证
4. 替换完成后，在 ESLint 或 CI 中添加检查规则防止回归

### 1.4 验收标准

- [ ] `features/` 目录下不存在直接调用 `fetch()` 的代码（SSE 场景使用 `authedStreamFetch`）
- [ ] 所有 API 调用统一经过认证链
- [ ] 手动 smoke 验证：工作流创建、AI 聊天、工作流执行、工作流同步均正常

## 2. S2.2 ESLint Feature 隔离规则

### 2.1 问题描述

`02-模块边界规范.md` 规定 Feature 之间不直接导入内部组件，但没有自动化检查机制。

### 2.2 实施方案

在前端 ESLint 配置中添加 `import/no-restricted-paths` 规则：

```json
{
  "rules": {
    "import/no-restricted-paths": ["error", {
      "zones": [
        {
          "target": "./src/features/admin/",
          "from": "./src/features/workflow/",
          "message": "admin 不能直接导入 workflow 内部模块"
        },
        {
          "target": "./src/features/workflow/",
          "from": "./src/features/admin/",
          "message": "workflow 不能直接导入 admin 内部模块"
        },
        {
          "target": "./src/features/auth/",
          "from": "./src/features/workflow/",
          "message": "auth 不能直接导入 workflow 内部模块"
        },
        {
          "target": "./src/features/workflow/",
          "from": "./src/features/auth/",
          "message": "workflow 不能直接导入 auth 内部模块"
        },
        {
          "target": "./src/features/knowledge/",
          "from": "./src/features/workflow/",
          "message": "knowledge 不能直接导入 workflow 内部模块"
        },
        {
          "target": "./src/features/settings/",
          "from": "./src/features/workflow/",
          "message": "settings 不能直接导入 workflow 内部模块"
        }
      ]
    }]
  }
}
```

### 2.3 备选方案（如 ESLint 插件不可用）

在 CI 中添加简单的 grep 检查：

```bash
# 检查 features 之间是否存在跨域导入
for feature in admin workflow auth knowledge settings; do
  for other in admin workflow auth knowledge settings; do
    if [ "$feature" != "$other" ]; then
      if grep -r "from '@/features/$other" frontend/src/features/$feature/ 2>/dev/null; then
        echo "VIOLATION: features/$feature imports from features/$other"
        exit 1
      fi
    fi
  done
done
```

### 2.4 操作步骤

1. 检查当前 ESLint 配置，确认是否已安装 `eslint-plugin-import`
2. 如已安装，添加 `import/no-restricted-paths` 规则
3. 如未安装，先用 CI grep 方案兜底
4. 运行一次检查，确认当前代码无违规（如有违规先修复）

### 2.5 验收标准

- [ ] ESLint 或 CI 中存在 Feature 隔离检查
- [ ] 当前代码通过检查（无跨 Feature 导入）
- [ ] 新增违规会被自动拦截

## 4. S2.4 CI 基础 workflows

### 4.1 当前状态

`.github/workflows/` 目录为空，没有任何 CI 配置。

### 4.2 需要创建的 workflows

#### 4.2.1 后端 CI

```yaml
# .github/workflows/ci-backend.yml
name: Backend CI
on:
  push:
    paths: ['backend/**']
  pull_request:
    paths: ['backend/**']

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r backend/requirements.txt
      - name: Lint
        run: |
          pip install ruff
          ruff check backend/
      - name: Type check
        run: |
          pip install pyright
          pyright backend/
```

#### 4.2.2 前端 CI

```yaml
# .github/workflows/ci-frontend.yml
name: Frontend CI
on:
  push:
    paths: ['frontend/**']
  pull_request:
    paths: ['frontend/**']

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install
        run: npm ci
        working-directory: frontend
      - name: Lint
        run: npm run lint
        working-directory: frontend
      - name: Type check
        run: npx tsc --noEmit
        working-directory: frontend
```

#### 4.2.3 Agent CI

```yaml
# .github/workflows/ci-agents.yml
name: Agents CI
on:
  push:
    paths: ['agents/**']
  pull_request:
    paths: ['agents/**']

jobs:
  test-agents:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        agent: [code-review-agent]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install
        run: pip install -r agents/${{ matrix.agent }}/requirements.txt
      - name: Test
        run: pytest agents/${{ matrix.agent }}/tests/ -q
```

### 4.3 操作步骤

1. 创建三个 workflow 文件
2. 确认 `backend/requirements.txt` 和 `frontend/package.json` 中的依赖完整
3. 推送后验证 CI 是否正常触发

### 4.4 验收标准

- [ ] `.github/workflows/` 下存在 3 个 workflow 文件
- [ ] 后端 CI 在 `backend/**` 变更时触发
- [ ] 前端 CI 在 `frontend/**` 变更时触发
