## 问题一：退出登录 & 删除工作流缺少二次确认

### 1.1 现状分析

**退出登录**

- 触发入口：`SidebarActivityBar.tsx` 底部的退出按钮
- 当前行为：点击后直接调用 `logoutAndRedirect()`，无任何确认
- `logoutAndRedirect` 定义在 `use-sidebar-navigation.ts`：
  ```ts
  const logoutAndRedirect = useCallback(async () => {
    await logout();        // 调用 auth.service 清除 session
    router.push('/login'); // 直接跳转登录页
  }, [router]);
  ```
- 风险：用户误触退出按钮后立即丢失当前会话，无法撤回

**删除工作流**

- 触发入口：侧边栏工作流列表右键菜单 → "删除"
- 当前行为：`use-workflow-sidebar-actions.ts` 中的 `onDeleteWorkflow` 已使用 `window.confirm()` 做了确认
  ```ts
  const confirmed = window.confirm(
    `确认删除工作流"${currentName}"？该操作不可恢复。`
  );
  ```
- 问题：`window.confirm()` 是浏览器原生弹窗，样式与项目 UI 风格不一致（白色系统弹窗 vs 暗色主题），且无法自定义按钮文案和样式

### 1.2 实施规划

**方案：创建统一的 ConfirmDialog 组件**

创建一个通用的确认对话框组件 `frontend/src/components/ui/ConfirmDialog.tsx`，替代 `window.confirm()` 并用于退出登录场景。

组件接口设计：
```ts
interface ConfirmDialogProps {
  open: boolean;
  title: string;
  description: string;
  confirmLabel?: string;       // 默认 "确认"
  cancelLabel?: string;        // 默认 "取消"
  variant?: 'danger' | 'default';  // danger 时确认按钮为红色
  onConfirm: () => void;
  onCancel: () => void;
}
```

**退出登录改造**

涉及文件：
- `frontend/src/components/layout/SidebarActivityBar.tsx`
  - 新增 `showLogoutConfirm` state
  - 退出按钮 onClick 改为 `setShowLogoutConfirm(true)`
  - 渲染 `<ConfirmDialog>` 组件
  - 确认后调用 `logoutAndRedirect()`

确认弹窗内容：
- 标题："确认退出登录"
- 描述："退出后需要重新登录才能访问工作流和学习数据。"
- 确认按钮："退出登录"（danger 样式）

**删除工作流改造**

涉及文件：
- `frontend/src/features/workflow/hooks/use-workflow-sidebar-actions.ts`
  - 移除 `window.confirm()` 调用
  - 改为返回一个 pending 状态，由调用方控制 ConfirmDialog 显示
- `frontend/src/components/layout/Sidebar.tsx`
  - 新增 `deleteConfirm` state（存储待删除的 workflowId + name）
  - `handleDelete` 改为设置 state 而非直接调用
  - 渲染 `<ConfirmDialog>` 组件
  - 确认后调用实际删除逻辑

确认弹窗内容：
- 标题："确认删除工作流"
- 描述：`确认删除工作流"${name}"？该操作不可恢复，所有节点和执行记录将被永久删除。`
- 确认按钮："删除"（danger 样式）

**涉及文件清单**

| 文件 | 操作 |
|------|------|
| `components/ui/ConfirmDialog.tsx` | 新建 |
| `components/layout/SidebarActivityBar.tsx` | 修改（退出确认） |
| `features/workflow/hooks/use-workflow-sidebar-actions.ts` | 修改（移除 window.confirm） |
| `components/layout/Sidebar.tsx` | 修改（删除确认 UI） |

## 问题二：AI 对话面板 — 模型选择器位置 & 思考深度选择器异常

### 2.1 现状分析

**模型选择器位置**

当前布局（`SidebarAIPanel.tsx`）：
```
┌─────────────────────────────────┐
│ AI 对话  [模型选择器▼]  [历史▼][+][⋯] │  ← 顶部 header
├─────────────────────────────────┤
│                                 │
│         对话消息区域              │
│                                 │
├─────────────────────────────────┤
│ ┌─────────────────────────────┐ │
│ │        输入文本框             │ │
│ ├─────────────────────────────┤ │
│ │ [模式▼] [深度]    [状态][→] │ │  ← 底部工具栏
│ └─────────────────────────────┘ │
└─────────────────────────────────┘
```

用户期望的布局（参考截图中 Kiro 的设计）：
- 模型选择器应放在输入框底部工具栏的右侧、发送按钮的左侧
- 顶部 header 只保留标题 + 历史/新建/更多按钮

**模型选择器当前实现细节**

- `SidebarAIPanel.tsx` 第 170-178 行：`<ModelSelector>` 渲染在顶部 header 的 `flex items-center gap-2` 容器中
- `ModelSelector.tsx`：支持两种模式（ChatModelSelectorProps / LegacyModelSelectorProps），当前 AI 面板使用 Chat 模式
- 选择器按钮样式：`rounded-lg border-[1.5px] px-2.5 py-1.5 text-[11px]`，紧凑设计，适合放在底部工具栏

**思考深度选择器异常**

- `ChatInputBar.tsx` 第 131-139 行：思考深度按钮的渲染条件是 `selectedModel.supportsThinking`
  ```tsx
  {selectedModel.supportsThinking && (
    <>
      <div className="h-3 w-px bg-border/30 mx-1" />
      <button type="button" onClick={cycleDepth} ...>
        <Route className="h-3 w-3" />{DEPTH_CONFIG[thinkingDepth].label}
      </button>
    </>
  )}
  ```
- **根因分析**：`ChatInputBar` 接收的 `selectedModel` prop 来自 `useAIChatStore` 的 `selectedModel` 字段
- `useAIChatStore` 的 `selectedModel` 初始值是 `DEFAULT_MODEL`（即 `FALLBACK_AI_MODEL_OPTIONS[0]`）
- `DEFAULT_MODEL` 是 DeepSeek V3，其 `supportsThinking: false`
- **关键问题**：`SidebarAIPanel` 中有两套模型状态：
  1. `selectedModel`（来自 `useAIChatStore`）— 类型 `AIModelOption`，用于 Track B（工作流节点），传给 `ChatInputBar`
  2. `selectedChatModel`（本地 state）— 类型 `ChatModelOption`，用于 Track A（对话面板），传给 `ModelSelector`
- 用户在顶部选择的模型更新的是 `selectedChatModel`（Track A），但 `ChatInputBar` 读取的是 `selectedModel`（Track B），两者不同步
- 因此即使用户选择了支持思考的模型（如 DeepSeek R1），`ChatInputBar` 仍然看到 `DEFAULT_MODEL.supportsThinking === false`，思考深度按钮永远不显示

**思考深度对后端的影响**

- `use-stream-chat.ts` 发送请求时：`thinking_level: thinkingDepth`
- 后端根据 `thinking_level` 决定是否启用思考模式
- 当 `thinkingDepth === 'fast'` 时，应该不开启思考（即不发送 `<think>` 标签请求）

### 2.2 实施规划

**模型选择器位置调整**

涉及文件：
- `frontend/src/components/layout/sidebar/SidebarAIPanel.tsx`
  - 从顶部 header 移除 `<ModelSelector>` 组件
  - 将 `selectedChatModel`、`chatModels`、`setSelectedChatModel`、`userTier`、`isModelsLoading`、`modelsError`、`fetchChatModels` 通过 props 传递给 `ChatInputBar`
- `frontend/src/components/layout/sidebar/ChatInputBar.tsx`
  - 新增 props 接收模型选择器相关数据
  - 在底部工具栏右侧（发送按钮左侧）渲染 `<ModelSelector>`

调整后布局：
```
┌─────────────────────────────────┐
│ AI 对话           [历史▼][+][⋯] │  ← 顶部 header（无模型选择器）
├─────────────────────────────────┤
│                                 │
│         对话消息区域              │
│                                 │
├─────────────────────────────────┤
│ ┌─────────────────────────────┐ │
│ │        输入文本框             │ │
│ ├─────────────────────────────┤ │
│ │ [模式▼][深度] [模型▼][状态][→]│ │  ← 底部工具栏
│ └─────────────────────────────┘ │
└─────────────────────────────────┘
```

**修复思考深度选择器不显示**

根因：`ChatInputBar` 使用 `selectedModel`（Track B 的 DEFAULT_MODEL）判断 `supportsThinking`，而非用户实际选择的 Chat 模型。

修复方案：
- `ChatInputBar` 的 props 中新增 `chatModel: ChatModelOption | null`
- 思考深度按钮的显示条件改为基于 `chatModel?.supportsThinking`
- **已确认**：`ChatModelOption`（定义在 `ai-catalog.service.ts`）当前没有 `supportsThinking` 字段
  ```ts
  // 当前 ChatModelOption 字段：
  // key, displayName, requiredTier, sortOrder, brandColor,
  // description, hasFallback, isRecommended, isPremium, isAccessible, skuId
  ```
- 需要在两处补充：
  1. 后端 `/api/ai/chat/models` 端点（`backend/app/api/ai_chat_models.py`）：通过 `get_sku_by_id()` 查询主 SKU 的 `supports_thinking` 字段，加入返回数据
     - SKU 数据库表已有 `supports_thinking` 字段（`backend/app/models/ai_catalog.py`）
     - `ai_catalog_service.get_sku_by_id()` 已查询此字段
     - 只需在 `models.append({...})` 中加入 `"supportsThinking": primary_sku.supports_thinking if primary_sku else False`
  2. 前端 `ChatModelOption` 类型（`frontend/src/services/ai-catalog.service.ts`）新增 `supportsThinking: boolean`

涉及文件：
- `frontend/src/services/ai-catalog.service.ts` — 检查/补充 `ChatModelOption.supportsThinking`
- `frontend/src/components/layout/sidebar/ChatInputBar.tsx` — 修改 props 和条件判断
- `frontend/src/components/layout/sidebar/SidebarAIPanel.tsx` — 传递正确的模型数据

**快速模式不开启思考**

当前 `thinkingDepth === 'fast'` 时，后端收到 `thinking_level: 'fast'`。需确认后端逻辑：
- 检查 `backend/app/services/` 中处理 `thinking_level` 的代码
- 确保 `fast` 模式下不请求模型的思考能力（不发送 `enable_thinking` 或等效参数）
- 如果后端已正确处理，则无需改动；如果没有，需要在 AI Router 层增加判断

**涉及文件清单**

| 文件 | 操作 |
|------|------|
| `components/layout/sidebar/SidebarAIPanel.tsx` | 修改（移除顶部模型选择器，传递 props） |
| `components/layout/sidebar/ChatInputBar.tsx` | 修改（新增模型选择器渲染 + 修复思考深度条件） |
| `services/ai-catalog.service.ts` | 检查/修改（ChatModelOption 类型） |
| 后端 AI 路由相关文件 | 检查（thinking_level=fast 的处理） |

## 问题三：侧边栏缩小时内容溢出/挤压

### 3.1 现状分析

**截图中的问题**

从用户提供的截图可以看到：右侧执行面板（RightPanelContent）在宽度较窄时，内容被严重挤压：
- "已完成" 状态标签被挤成竖排（"已/完/成"）
- 步骤标题和线路信息被挤压换行
- 左侧 Activity Bar 的图标在窄屏下仍然全部显示，占据了大量空间

**左侧 Activity Bar 分析**

`SidebarActivityBar.tsx` 当前结构：
```
┌──────┐
│ 执行 │  ← rightPanelDockedToSidebar 时显示
│ 用户 │
│ ──── │  ← 分隔线
│ AI   │  ← IMMOVABLE_UPPER
│ 商店 │
│ 工作流│  ← dynamicPinned（用户可 pin/unpin）
│ 知识库│
│ 样例 │
│ 仪表盘│
│ 拓展 │
│      │  ← flex-1 弹性空间
│ 升级 │
│ 钱包 │
│ 设置 │
│ 手册 │
│ 退出 │
└──────┘
```

- Activity Bar 固定宽度 `w-12`（48px），不会缩小
- 所有图标按钮 `h-10 w-10`，垂直排列
- 当图标数量多时（最多 14 个按钮），在小屏幕上可能超出可视区域
- 当前没有任何溢出处理机制（无 `overflow-y-auto`，无折叠逻辑）

**左侧面板分析**

`Sidebar.tsx` 中面板容器：
```tsx
<div className="hidden flex-col border-l border-border bg-background lg:flex"
     style={{ width: leftPanelWidth }}>
```
- `lg:flex` 意味着只在 `≥1024px` 时显示面板内容
- `leftPanelWidth` 可调整，范围 `200px ~ 480px`
- 面板内容没有针对窄宽度的响应式处理

**右侧面板分析**

`RightPanel.tsx`：
- `rightPanelWidth` 范围 `240px ~ 520px`
- 面板内容（`RightPanelContent.tsx`）使用 `grid-cols-2` 布局显示状态卡片
- 执行步骤列表（`ExecutionTraceList` → `TraceStepItem`）中的状态标签和标题在窄宽度下被挤压
- 没有针对面板宽度的响应式断点处理

**核心问题**

1. Activity Bar 图标过多时没有滚动或折叠机制
2. 右侧执行面板的内容（特别是状态标签、步骤标题）在窄宽度下没有适配
3. 左右面板的最小宽度（200px / 240px）可能仍然不足以正常显示某些内容

### 3.2 实施规划

**方案 A：Activity Bar 溢出处理**

涉及文件：`frontend/src/components/layout/SidebarActivityBar.tsx`

1. 给 Activity Bar 主容器添加 `overflow-y-auto scrollbar-hide`，防止图标超出屏幕
2. 当可用高度不足时（通过 `ResizeObserver` 检测），将部分低优先级图标收入 "更多" 菜单：
   - 高优先级（始终显示）：用户面板、AI 对话、节点商店、设置、退出
   - 中优先级（空间不足时收入菜单）：dynamicPinned 面板（工作流、知识库、样例、仪表盘）
   - 低优先级（最先收入菜单）：拓展、钱包、手册、升级

实现思路：
```tsx
const [visibleCount, setVisibleCount] = useState(Infinity);
// ResizeObserver 监听容器高度
// 计算可容纳的按钮数量 = Math.floor((containerHeight - fixedHeight) / 44)
// fixedHeight = 上下固定区域高度（用户面板 + 分隔线 + 底部按钮组）
// 超出部分渲染为 "⋯" 按钮 → 点击弹出 popover 菜单
```

**方案 B：右侧执行面板窄宽度适配**

涉及文件：
- `frontend/src/components/layout/sidebar/RightPanelContent.tsx`
- `frontend/src/features/workflow/components/execution/TraceStepItem.tsx`

1. 状态总览卡片：当面板宽度 < 300px 时，`grid-cols-2` 改为 `grid-cols-1`
   - 可通过 CSS container query 或传入 width prop 实现
2. 执行步骤中的状态标签：
   - 当前 "已完成" 标签使用固定宽度的圆形/方形 badge
   - 改为使用 `whitespace-nowrap` + `overflow-hidden text-ellipsis` 防止竖排
   - 或在窄宽度下只显示图标（✓/●/✕）不显示文字
3. 步骤标题和线路信息：
   - 添加 `truncate` 或 `line-clamp-2` 防止过度换行
   - 线路标签改为横向滚动或折叠显示

**方案 C：面板最小宽度保护**

涉及文件：`frontend/src/stores/use-panel-store.ts`

- 考虑将 `RIGHT_PANEL_MIN` 从 240px 提升到 280px
- 或在内容层面做好 240px 的适配（方案 B）

**方案 D：左侧面板内容窄宽度适配**

涉及文件：各面板组件

- 当 `leftPanelWidth < 260px` 时，部分面板的标题栏按钮可以收入菜单
- AI 对话面板的顶部 header 在窄宽度下可以将历史/新建/更多按钮收入单个菜单

**涉及文件清单**

| 文件 | 操作 |
|------|------|
| `components/layout/SidebarActivityBar.tsx` | 修改（溢出处理 + 折叠菜单） |
| `components/layout/sidebar/RightPanelContent.tsx` | 修改（窄宽度响应式） |
| `features/workflow/components/execution/TraceStepItem.tsx` | 修改（状态标签适配） |
| `stores/use-panel-store.ts` | 可选修改（最小宽度调整） |

| 优先级 | 问题 | 预估工作量 | 理由 |
|--------|------|-----------|------|
