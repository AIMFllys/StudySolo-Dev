## 一、问题背景

大语言模型的普及使 AI 辅助学习成为可能，但现有工具存在一个根本局限：**单轮对话无法覆盖完整的学习过程**。

真实的学习任务往往是多阶段的。以"系统学习机器学习"为例，完整流程包含目标拆解、大纲生成、内容提取、知识总结、记忆巩固、效果测验等多个环节，各环节之间存在明确的数据依赖关系。当前主流方案的局限性如下：

| 现有工具 | 核心局限 |
|---------|---------|
| 通用聊天助手（ChatGPT / DeepSeek 等） | 逐轮对话，上一步产出无法自动传入下一步；无法编排多步骤流程 |
| 通用工作流平台（Dify / Coze 等） | 面向通用场景，缺乏学习领域专用节点；搭建门槛高，用户须理解 DAG 连线逻辑 |
| 手动拼接多个 AI 工具 | 效率低下，流程不可复用，学习过程不可观测 |

**StudySolo 针对上述痛点而设计**：一个专注于学习场景、以自然语言驱动、可视化编排、支持社区共建的 AI 智能体平台。

## 二、需求分析

### 2.1 核心用户场景

平台面向大学生及终身学习者，核心诉求是：**不想学复杂工具，但希望 AI 能帮我把一个完整的学习任务自动化地拆解并执行下来**。

### 2.2 为什么不只做一个智能体

单一智能体可以完成单步任务，但无法胜任跨步骤的学习流程。真实学习中，每个环节的目标、模型选择、输出格式要求各不相同，且存在数据依赖。更重要的是：

- 不同学科、不同用途需要**不同的流程组合**，不能一刀切；
- 学习流程需要**可保存、可复用、可分享**，而非每次从零开始；
- 用户需要看到**每一步 AI 在做什么**，建立对工具的理解和信任。

这要求平台提供的不是一个固定的智能体，而是一套**让用户能够创建、运行并管理自己的智能体流程的开放平台**。

### 2.3 具体需求

**需求一：自然语言描述学习目标，自动生成工作流**
用户无需理解工作流搭建逻辑，只需在侧边栏说出学习需求，系统即可生成包含多个步骤的可视化工作流。未来将进一步强化左侧 AI 面板能力，支持用户以对话方式完全配置工作流的每个细节。

**需求二：可视化编辑与灵活调整**
生成的工作流以画布节点形式展示，支持拖动、增删、连线。用户可在 AI 生成的基础上自由修改，满足个性化需求。

**需求三：分步流式执行，全程可观测**
执行时各节点按依赖顺序逐步运行，输入、输出、状态、耗时全程实时可见，不是黑盒。

**需求四：工作流与节点的社区共享**
优秀工作流和节点应当可被复用和共享，形成社区知识资产，降低学习流程设计成本。

**需求五：平台能力持续扩展**
节点类型、模型支持、外部工具对接需要随时间持续增加，平台本身需为此提供标准化的扩展机制。

## 三、智能体架构

### 3.1 核心抽象：节点即智能体

StudySolo 的核心设计思想是：**每个工作流节点本身就是一个独立的智能体**。

每个节点拥有：
- 独立的 System Prompt（定义该智能体的角色与行为约束）
- 独立的模型路由配置（可按任务特性选择最优大模型）
- 独立的输入/输出契约（通过 ExecutionContext 与其他节点交换数据）
- 独立的配置元数据（通过 config_schema 声明可调参数）

多个节点通过有向无环图（DAG）连接，本质上构成**多智能体协作系统**。上游智能体的输出作为下游智能体的输入，由 DAG 执行引擎按拓扑序协调调度。

### 3.2 四层运行架构

```
Layer 0 ── 意图理解层（Intent Classifier）
  │  用户自然语言输入 → AI 分类为 BUILD / MODIFY / CHAT / ACTION
  ▼
Layer 1 ── 规划层（AI Planner）
  │  将学习目标拆解为节点图（含节点类型、连线、初始配置）
  ▼
Layer 2 ── 编排层（DAG Orchestrator）
  │  拓扑排序确定执行顺序，处理分支 / 合并 / 循环逻辑
  ▼
Layer 3 ── 执行层（18 种 Node Agent）
  │  各节点独立执行，通过 SSE 流式回传进度与结果
  ▼
Layer 4 ── 路由层（AI Model Router）
     8 个 AI 平台 × 3 种路由策略，自动选择最优模型，多级容灾降级
```

### 3.3 侧边栏 AI 面板的三种交互模式

| 模式 | 能力 | 输出格式 |
|------|------|---------|
| 对话模式 | 回答学习问题，不操作画布 | Markdown 流式文本 |
| 规划模式 | 分析当前工作流结构，给出改进建议 | XML 结构化建议（含优先级） |
| 创建模式 | 直接操作画布——新增/删除/修改节点与连线 | JSON 操作指令 |

**创建模式的关键机制**：AI 能够"看见"当前画布。前端在每次请求时将画布上所有节点的类型、标签、状态、上下游关系及坐标序列化为结构化文本，注入大模型 System Prompt。模型返回的操作指令由 ActionExecutor 解释执行，并支持快照式撤销回滚，确保安全性。

### 3.4 节点体系（当前 18 种）

| 类别 | 节点 | 功能 |
|------|------|------|
| **输入类** | trigger_input | 工作流入口，承载用户原始学习目标 |
| | knowledge_base | 知识库节点，支持文件上传与文档内容注入 |
| | web_search | 联网搜索节点，多源并发检索 |
| **分析类** | ai_analyzer | 多维度深度分析 |
| | ai_planner | 将目标分解为结构化学习方案 |
| | content_extract | 从长文本中抽取关键信息 |
| | compare | 多角度对比分析 |
| **生成类** | outline_gen | 大纲生成 |
| | summary | 总结归纳 |
| | flashcard | 闪卡生成（JSON 结构化输出） |
| | quiz_gen | 测验生成（JSON 结构化输出） |
| | mind_map | 思维导图生成 |
| | merge_polish | 多源合并与润色 |
| **交互类** | chat_response | 对话回复 |
| **输出类** | export_file | 文件导出（Markdown / TXT 下载） |
| | write_db | 结果持久化写入数据库 |
| **控制流** | logic_switch | 条件分支 |
| | loop_group | 循环容器 |

### 3.5 多平台 AI 模型路由

后端 AI Router 对接 8 个主流大模型平台，实现统一调度与多级容灾：

| 平台 | 代表模型 | 接入方式 |
|------|---------|---------|
| DeepSeek | V3 / R1 | 原生 |
| 通义千问（百炼） | Turbo / Plus / Max / Qwen3-Max | 原生 + 七牛代理 |
| 智谱 GLM | Flash / GLM-4 / GLM-4.5 / GLM-4.7 | 原生 |
| 豆包（火山引擎） | Pro-32K / Pro-256K / Seed 2.0 | 原生 + 七牛代理 |
| 月之暗面 Kimi | 8K / 128K / K2 / K2.5 | 原生 + 七牛代理 |
| 七牛云 | 多厂商代理通道 | 代理 |
| 硅基流动 | Qwen2.5-72B | 代理 |
| 火山引擎 | Doubao 系列 | 原生 |

支持 3 种路由策略：`native_first`（原生优先）、`proxy_first`（代理优先降成本）、`capability_fixed`（能力固定）。每个节点类型配置 2-3 个 SKU 候选，单平台宕机自动切换。

## 四、技术方案

### 4.1 整体技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端框架** | Next.js 15 + React 19 + TypeScript | App Router，SSR + CSR 混合渲染 |
| **前端样式** | Tailwind CSS v4 | CSS-first 配置，设计令牌统一管理 |
| **画布引擎** | @xyflow/react（React Flow） | 工业级节点画布，支持拖拽、连线、分组、缩放 |
| **前端状态** | Zustand | 轻量级状态管理，内置 Undo/Redo 快照 |
| **后端框架** | Python + FastAPI | 高性能异步框架，原生支持 SSE 流式推送 |
| **执行引擎** | 自研 DAG Executor | 拓扑排序 + ExecutionContext 黑板模型 |
| **数据库** | Supabase (PostgreSQL) + Row Level Security | 多租户数据隔离，RLS 全覆盖 |
| **认证** | Supabase Auth + JWT + 自定义验证码 | Canvas 拼图验证码 + IP 登录锁定 |
| **实时通信** | Server-Sent Events（SSE） | 节点执行进度流式推送至前端 |
| **生产部署** | 阿里云 ECS + Nginx + Baota 面板 | 当前已完成生产环境部署并上线 |

### 4.2 DAG 执行引擎

执行引擎（`backend/app/engine/executor.py`）是平台核心，负责将工作流图转化为可执行任务序列：

1. 从 POST 请求体直接获取当前画布的 nodes 与 edges，确保执行始终对齐用户眼前的版本；
2. 拓扑排序确定执行顺序；
3. 逐节点执行，通过 ExecutionContext 在节点间传递中间结果；
4. 每个节点通过 SSE 推送 7 种事件：`node_input`、`node_status`、`node_token`、`node_done`、`loop_iteration`、`save_error`、`workflow_done`。

前端执行面板实时渲染运行状态、流式输出与链路追踪，支持按 root-to-leaf 分支筛选查看。

### 4.3 工作流数据同步

采用双缓冲策略：
- **本地缓冲**：IndexedDB 快照 + 300ms 防抖，保障离线可恢复；
- **云端同步**：Supabase 数据库 + 5s 节流，保障多端一致；
- 执行时直接提交内存中的当前图结构，杜绝"编辑态与执行态不一致"问题。

### 4.4 Prompt 模块化系统

所有 AI 提示词采用 Markdown 文件 + 模板变量架构，而非硬编码在代码中：

```
backend/app/prompts/
├── identity.md              ← 全局身份设定（所有模式共享）
├── intent_classifier.md     ← 意图分类器
├── mode_chat / plan / create.md  ← 三种面板模式
└── nodes/_base_prompt.md    ← 节点通用约束

backend/app/nodes/<category>/<node_type>/
└── prompt.md                ← 每个节点独立的 System Prompt
```

提示词通过 `prompt_loader.py` 统一加载，支持 `{{变量}}` 渲染和 LRU 缓存，可独立迭代并通过 Git 版本管理。

## 五、创新亮点

### 亮点一：自然语言即工作流

用户无需了解 DAG、节点、连线等概念，侧边栏直接用自然语言描述目标，系统完成从意图识别到工作流生成的全链路：
- **从零构建**：描述学习目标 → AI 自动生成完整节点图；
- **增量修改**：说"在总结后面加一个闪卡" → AI 直接操作画布完成；
- **规划建议**：切换规划模式 → AI 分析工作流结构并给出改进方案。

这是本平台最核心的体验差异：用户始终在用自然语言与平台沟通，而不是拖拽配置。

### 亮点二：面向完整平台而非单一智能体

本项目并非一个单一智能体，而是一套**让用户能够创建、运行、共享和管理智能体流程的完整平台**。这一定位来自显而易见的需求逻辑：

- 不同学习目标需要不同的节点组合——单一智能体无法覆盖；
- 好的学习流程应当被保存和复用——单次对话不解决此问题；
- 平台型产品才能积累社区资产——每个用户的贡献都为后来者降低成本。

因此，平台同时具备：用户体系、工作流管理、社区共享、节点商店、管理后台、模型配置、计费体系等完整配套能力。**这不是过度开发，而是平台逻辑本就如此。**

### 亮点三：社区化工作流共享与分叉

工作流是可复用的学习资产：
- 用户可发布工作流至社区，供他人浏览、收藏、点赞；
- 支持分叉（Fork）——在他人工作流基础上修改形成自己的版本；
- 社区积累的工作流模板将持续降低学习流程设计门槛。

### 亮点四：开放式节点共建生态

平台预置 18 种节点，同时支持用户自行创建并发布提示词节点：
- 用户定义节点名称、Prompt、输出格式，可附带领域知识文件；
- JSON 输出格式支持 AI 辅助生成 Schema 约束，确保结构化输出质量；
- 发布后进入社区节点商店，其他用户可直接拖入工作流使用；
- 封装后的 Prompt 对使用者不可见，保护创作者知识产权。

**用户既是平台的使用者，也是平台能力的构建者。**

### 亮点五：SOP + Skills 标准化开发体系

为保证平台可持续演进，我们将"新增节点"这一关键开发动作标准化为完整 SOP 文档体系，并封装为 AI 开发技能（Skills）：

开源仓库：https://github.com/AIMFllys/StudySolo

| SOP 文档 | 路径 | 用途 |
|---------|------|------|
| 00-节点与插件分类判断.md | `docs/项目规范与框架流程/功能流程/新增AI工具/` | 先分类再开发 |
| A型-LLM提示词节点-SOP.md | 同上 | 后端节点包创建 / 路由配置 / 前端注册 / 全流程 Checklist |
| B型-外部工具节点-SOP.md | 同上 | 外部 API 对接规范 |
| 01-现有节点功能补全总则.md | 同上 | 存量节点能力审计与补全标准 |

得益于这套体系：**新增一个 A 型节点，后端不超过 3 个文件，前端不超过 4 个文件。** 初期框架搭建完成后，后续扩展成本极低，平台将持续增加新节点、新模型、新功能。

### 亮点六：完整的生产级治理能力

区别于纯原型级项目，平台当前已具备完整的生产运营配套：
- **管理后台**：10 个模块，涵盖用户管理、模型配置、SKU 管理、使用量统计、审计日志；
- **计费体系**：基于 SKU 的模型调用成本核算，支持会员分层（Free / Pro / Pro+ / Ultra）；
- **安全机制**：Canvas 拼图验证码、IP 锁定、Row Level Security、JWT 认证。

## 六、应用场景

**场景一：系统化学习新知识领域**
用户输入"帮我制定机器学习入门学习计划"，系统生成"目标拆解 → 大纲生成 → 内容提取 → 总结归纳 → 闪卡生成 → 测验评估"完整工作流，执行后可导出学习材料留存。

**场景二：课后复习与知识巩固**
上传课程 PDF 至知识库节点，连接"内容提取 → 闪卡生成"，快速将课件转化为记忆卡片，常用流程可保存为模板复用。

**场景三：社区共享学习流程**
擅长论文阅读的用户构建"摘要提取 → 方法论分析 → 创新点提炼 → 批判性评价"工作流并发布，其他用户直接分叉使用，无需从零设计。

**场景四：教师辅助教学设计**
教师构建"知识点讲解 → 随堂测验 → 错题分析 → 补充学习"工作流，部署后供学生自主使用，管理后台可查看使用统计。
