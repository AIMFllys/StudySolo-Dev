## 第一章 · 产品概述

### 1.1 产品定位

**StudySolo** 是 1037Solo 生态中的 AI 学习工作流平台，核心定位是：

> **把“学习目标”转换成可视化、可执行、可复用的 AI 工作流。**

它不是单纯的聊天助手，也不是把多个工具拼在一起的“操作台”。  
它更像是一张可交互的学习画布：用户先定义目标，再把拆解、检索、生成、总结、测验、导出这些步骤组织成一条工作流，系统负责按依赖顺序执行，并把结果流式返回到前端。

如果把传统问答式 AI 看作“即时回答器”，那么 StudySolo 更接近“学习过程编排器”：

- 它允许用户看见每一步在做什么。
- 它允许用户替换、插入、删减某个步骤。
- 它允许同一份工作流被重复运行、共享、收藏、协作和版本化使用。

### 1.2 我们究竟在解决什么问题？

| 痛点 | 传统方案的困境 | StudySolo 的解决方式 |
|------|----------------|----------------------|
| 学习路径不清晰 | 普通聊天式 AI 往往只回答当前问题，难以稳定地把复杂目标拆解成完整路径 | 用可视化工作流把输入、分析、检索、生成、输出串成清晰链路 |
| 多工具切换成本高 | 用户需要在聊天框、搜索、知识库、文档整理、导出工具之间反复切换 | 将这些能力收束为统一节点体系，在同一张画布上执行 |
| 过程不可控 | 黑盒式回答一旦偏题，用户很难定位是哪一步出错 | 每个节点都有输入、状态、输出，可以单独观察与调整 |
| 结果难复用 | 传统对话记录很难沉淀为稳定模板 | 工作流可以保存、共享、收藏、再运行、分叉协作 |
| 模型选择混乱 | 不同模型擅长不同任务，用户往往不知道该如何选 | 系统内置 catalog、路由策略与可选模型入口，支持按任务匹配模型 |

### 1.3 产品愿景

StudySolo 的目标不是替代人的学习判断，而是把学习过程中的重复劳动、结构化组织和内容生产部分交给系统完成。

> **让用户从“反复提示 AI”转向“设计自己的学习工作流”。**

在这个定位下，StudySolo 更强调三件事：

1. **可视化**：用户能够看到学习过程本身，而不仅是最终答案。
2. **可执行**：工作流不是静态流程图，而是可以真正运行的 DAG。
3. **可复用**：优秀工作流可以沉淀成模板，成为长期资产。

### 1.4 适用人群与典型场景

| 用户类型 | 使用方式 |
|------|------|
| 自主学习者 | 把一个课程目标拆成分析、总结、题库、闪卡、导出等多个步骤 |
| 研究型用户 | 把联网搜索、知识库检索、对比辨析和总结组合成研究链路 |
| 内容整理者 | 把原始材料提炼为大纲、摘要、导图、题库和导出文件 |
| 教学设计者 | 设计标准化工作流，重复生成不同章节或不同主题的学习材料 |
| 社区协作者 | 将工作流公开、收藏、点赞、分叉并邀请协作者共同维护 |

## 第二章 · 平台架构总览

StudySolo 采用前后端分离架构，前端负责工作流编排、画布交互和流式展示，后端负责 API、执行引擎、模型路由、知识库与用量统计，数据底座基于 Supabase。

### 2.1 前端技术栈

| 技术 | 当前版本 | 职责 |
|------|------|------|
| **Next.js** | `16.1.6` | App Router 路由、页面组织与前端应用框架 |
| **React** | `19.2.3` | 组件渲染与交互驱动 |
| **TypeScript** | `5` | 类型约束与业务契约 |
| **Tailwind CSS** | `v4` | 样式系统与页面设计实现 |
| **Zustand** | `5.0.11` | 工作流编辑状态、聊天状态与界面状态管理 |
| **@xyflow/react** | `12.10.1` | 工作流画布、节点与连线渲染 |
| **Framer Motion** | `12.38.0` | 动效与页面过渡 |

前端的核心不是“页面堆叠”，而是围绕工作流做分层：

- 路由与页面在 `frontend/src/app/`
- 业务功能在 `frontend/src/features/`
- 服务调用在 `frontend/src/services/`
- 全局状态在 `frontend/src/stores/`
- 类型契约在 `frontend/src/types/`

### 2.2 后端技术栈

| 技术 | 当前版本 | 职责 |
|------|------|------|
| **FastAPI** | `>=0.115,<1.0` | API 服务、异步请求处理与接口聚合 |
| **Pydantic** | `>=2.10,<3.0` | 请求响应模型与契约校验 |
| **Supabase Python** | `>=2.11,<3.0` | 数据访问与认证协作 |
| **OpenAI SDK** | `>=1.60,<2.0` | 兼容多家 OpenAI 风格模型调用 |
| **SSE-Starlette** | `>=2.2,<3.0` | 流式事件输出 |
| **SlowAPI** | `>=0.1.9,<1.0` | 限流与安全保护 |

后端结构以“执行”和“编排”为中心：

- `backend/app/api/`：统一聚合 `auth`、`workflow`、`ai`、`usage`、`admin` 等接口
- `backend/app/engine/`：DAG 执行与流式事件分发
- `backend/app/nodes/`：插件化节点实现
- `backend/app/services/`：AI 路由、catalog、usage、知识库等服务
- `backend/app/models/`：Pydantic 数据契约

### 2.3 数据与共享底座

- **数据库底座**：Supabase PostgreSQL
- **认证能力**：Supabase Auth
- **安全边界**：RLS（行级安全）
- **向量/知识库能力**：pgvector 相关能力由 Supabase 承载
- **共享层**：`shared/` 在本仓库中是 **Git submodule**

这里有一个非常重要的公开说明：

- 本仓库中的 `shared/` 是 **Git submodule**
- Platform Monorepo 中的 `StudySolo/` 是 **git subtree**
- 这两件事不是同一层同步关系，文档中不能混写

### 2.4 多模型 AI 引擎矩阵

根据当前 `backend/config.yaml`，StudySolo 接入了多家模型提供方与代理通道：

| Provider | 角色 | 说明 |
|------|------|------|
| `dashscope` | native | 阿里云百炼原生通道 |
| `deepseek` | native | DeepSeek 官方通道 |
| `moonshot` | native | 月之暗面通道 |
| `volcengine` | native | 火山引擎通道 |
| `zhipu` | native | 智谱 AI 通道 |
| `qiniu` | proxy | 七牛云代理通道 |
| `siliconflow` | proxy | 硅基流动代理通道 |
| `compshare` | proxy | 优云智算通道 |

模型目录体系已经从“单表散列”升级为 catalog 结构：

- 模型家族表：`ai_model_families`
- 模型 SKU 表：`ai_model_skus`
- 前后端正式主选型字段：`selected_model_key`
- 正式价格字段：`*_cny`

这意味着用户在界面中看到的模型选择，不再是松散的 provider/model 字符串，而是围绕 SKU、family、billing channel、tier 和显示策略组织。

### 2.5 智能路由与流式执行机制

StudySolo 的模型调用并不是简单地把请求发给某一个固定接口，而是按任务类型选择路由策略。

当前配置中可以看到这些典型模式：

- `native_first`：优先走原生提供方
- `proxy_first`：优先走代理通道
- `capability_fixed`：某类能力固定绑定专门服务

例如：

- `trigger_input`、`ai_analyzer`、`ai_planner` 等任务使用 `native_first`
- `merge_polish`、`premium_chat` 使用 `proxy_first`
- `search`、`ocr` 使用 `capability_fixed`

执行层还包含这些现实约束：

- 单次执行超时控制：`30000 ms`
- 最大重试次数：`3`
- 单工作流最大节点数：`12`
- JSON 校验重试次数：`3`

工作流运行后的输出通过 **SSE** 持续回流到前端，因此用户能够在画布或聊天侧栏实时看到节点状态变化和输出内容。

## 第三章 · 核心能力

### 3.1 动态生成式工作流

StudySolo 支持从一个学习目标出发，生成后续工作流骨架。  
用户输入的不一定是完整步骤，也可以只是一个目标，例如：

- “我想在一周内理解一篇论文的核心论点”
- “帮我把这组课程笔记整理成复习卡片和测验题”
- “基于知识库资料做一条可重复运行的考试准备工作流”

系统会将目标转化为一组节点，并按照依赖关系形成 DAG，后续用户可以继续编辑、删改或扩展。

### 3.2 可视化交互画布

画布是 StudySolo 的主界面能力，不只是“展示流程图”，而是真正的工作流编排器。

用户可以在画布中：

- 新增节点
- 删除节点
- 拖拽调整布局
- 建立顺序连线
- 查看节点输入输出
- 在执行后观察各步骤状态变化
- 将工作流保存为自己的资产

当前边类型只有一种：

- `sequential`

这意味着顺序依赖是工作流的基础表达方式，条件分流和循环并不是通过多种边类型表达，而是由节点与结构节点来承载。

### 3.3 插件化节点生态

节点体系是 StudySolo 的核心抽象。  
当前真实节点类型包括：

- 输入：`trigger_input`、`knowledge_base`、`web_search`
- 分析：`ai_analyzer`、`ai_planner`、`logic_switch`、`loop_map`
- 生成：`outline_gen`、`content_extract`、`summary`、`flashcard`、`compare`、`mind_map`、`quiz_gen`、`merge_polish`
- 交互：`chat_response`
- 输出：`write_db`、`export_file`
- 结构：`loop_group`

这套体系的意义在于，用户处理的不是一串无法维护的提示词，而是一组语义明确、职责清晰、可组合的节点。

### 3.4 精准上下文流转

StudySolo 的工作流并不是把所有历史拼接成一个超长上下文，而是让上下文沿着节点之间的关系流动。

这样做有几个直接好处：

- 每个节点只接收自己真正需要的上游信息
- 结果更容易解释，也更方便定位问题
- 节点输出可以复用到不同分支
- 大型工作流更容易维护

结合节点状态，用户可以更清楚地理解执行进度。当前节点状态包括：

- `pending`
- `running`
- `waiting`
- `done`
- `error`
- `skipped`
- `paused`

### 3.5 拓扑流式执行引擎

后端执行引擎会按 DAG 顺序调度节点，并将关键事件以 SSE 方式推送到前端。

当前前端对齐的主要 SSE 事件包括：

- `node_status`
- `node_input`
- `node_token`
- `node_done`
- `loop_iteration`
- `workflow_done`
- `save_error`

这让用户看到的不再只是“运行中”三个字，而是一个逐步推进的执行过程。

### 3.6 知识库、联网与导出能力

StudySolo 不把“学习资料”局限在聊天上下文里，而是把资料来源做成明确能力：

- **知识库节点**：从用户资料或项目资料中检索可复用内容
- **联网搜索节点**：从公开网页获取最新资料
- **导出节点**：将整理后的内容导出为文件
- **写库节点**：把结构化结果沉淀到数据库或工作流结果记录中

这使工作流既能处理“已有材料”，也能处理“需要外部补充”的目标。

### 3.7 协作、公开页与市场能力

从当前 API 组织可以看出，工作流已经不只是个人私有对象，它还具备公开与协作能力。

工作流相关接口分为几个层面：

- `workflow`：工作流 CRUD
- `workflow_execute`：运行与执行
- `workflow_social`：点赞、收藏、公开页、市场相关交互
- `workflow_collaboration`：协作者、邀请与共享相关能力

对用户来说，这代表：

- 工作流可以被公开展示
- 工作流可以进入市场或公共空间
- 工作流可以被收藏、点赞、分叉
- 团队或多人场景下可以继续向协作方向扩展

### 3.8 管理后台、反馈与用量分析

StudySolo 当前已经具备完整的项目级后台与治理接口分组：

- `admin_auth`
- `admin_dashboard`
- `admin_users`
- `admin_notices`
- `admin_workflows`
- `admin_models`
- `admin_members`
- `admin_ratings`
- `admin_config`
- `admin_audit`

同时，还存在两类与平台运营直接相关的接口：

- `feedback`：用户反馈与奖励入口
- `usage`：用量、成本与统计视角

用量体系的公开口径已经切换到人民币字段：

- 总成本、实时成本、按分钟聚合等字段统一使用 `*_cny`
- 调用维度中包含 `sku_id`、`family_id`、`billing_channel`

这让 StudySolo 不只是“能跑起来”，而是具备持续运营、观察和治理的基础。

## 第四章 · 节点功能大全

### 4.1 节点体系概述

StudySolo 当前的节点体系可分为六大类：

```text
输入类 (Input)        -> 将目标、知识库或外部搜索结果引入工作流
分析类 (Analysis)     -> 对信息进行拆解、判断、规划与控制流处理
生成类 (Generation)   -> 产出大纲、总结、闪卡、题库、导图等内容
交互类 (Interaction)  -> 面向用户给出回复与引导
输出类 (Output)       -> 将结果写入数据库或导出文件
结构类 (Structure)    -> 对循环区域进行容器化组织
```

这套节点不是装饰性的图标集合，而是工作流执行与前端呈现共同依赖的核心语义层。

### 4.2 输入类节点 (Input)

#### ▶️ 触发输入节点 · Trigger Input

**类型**：`trigger_input`  
**作用**：接收用户目标，作为工作流起点。

适用场景：

- 用户用一句话定义学习任务
- 工作流需要一个统一入口参数
- 为后续分析与规划节点提供原始上下文

典型输出：

- 学习目标
- 输入约束
- 主题范围

#### 📚 知识库检索节点 · Knowledge Base

**类型**：`knowledge_base`  
**作用**：从知识库中检索与当前目标相关的内容。

适用场景：

- 基于已有课程资料继续加工
- 对内部文档、上传资料进行语义检索
- 将已有知识沉淀纳入标准工作流

典型输出：

- 相关片段列表
- 主题相关摘要
- 上游生成所需的素材池

#### 🌐 联网搜索节点 · Web Search

**类型**：`web_search`  
**作用**：连接公开网络内容，为工作流补充最新资料。

适用场景：

- 需要新近资料、网页信息或外部背景
- 研究型任务需要更多上下文来源
- 补足知识库中不存在的内容

典型输出：

- 网页结果摘要
- 可供分析与总结的搜索材料

### 4.3 分析类节点 (Analysis)

#### 🔍 AI 分析器节点 · AI Analyzer

**类型**：`ai_analyzer`  
**作用**：对用户目标或上游材料进行需求拆解和重点提取。

适用场景：

- 识别任务边界
- 明确学习重点
- 提取目标中的隐含约束

#### 📐 AI 规划器节点 · AI Planner

**类型**：`ai_planner`  
**作用**：生成后续执行步骤与整体工作流策略。

适用场景：

- 从“目标”转为“步骤”
- 明确哪些节点应该出现
- 为下游生成类节点提供顺序基础

#### 🔀 逻辑分支节点 · Logic Switch

**类型**：`logic_switch`  
**作用**：根据条件决定后续走向。

适用场景：

- 根据输入特征选择不同处理分支
- 不同难度、不同资料来源、不同输出要求使用不同链路

说明：

- 当前工作流边类型仍然只有 `sequential`
- 分支能力主要依赖 `logic_switch` 节点自身来承载

#### 🔄 循环映射节点 · Loop Map

**类型**：`loop_map`  
**作用**：对列表、批量对象或多项输入逐项处理。

适用场景：

- 逐章处理学习材料
- 逐题生成解释
- 批量整理多份资料

### 4.4 生成类节点 (Generation)

#### 📋 大纲生成节点 · Outline Gen

**类型**：`outline_gen`  
**作用**：把复杂材料整理成层级清晰的大纲。

适用场景：

- 课程结构化整理
- 章节组织
- 形成后续总结与写作骨架

#### 🔬 内容提炼节点 · Content Extract

**类型**：`content_extract`  
**作用**：提取关键概念、事实和重点信息。

适用场景：

- 长文拆解
- 材料精炼
- 高密度知识压缩

#### 📝 学习总结节点 · Summary

**类型**：`summary`  
**作用**：将上游内容聚合为结构化总结。

适用场景：

- 复习摘要
- 章节总结
- 统一输出终稿前的中间收束

#### 🃏 闪卡生成节点 · Flashcard

**类型**：`flashcard`  
**作用**：生成适合记忆训练的卡片内容。

适用场景：

- 定义-概念型学习
- 术语复习
- 章节记忆巩固

#### 📝 测验出题节点 · Quiz Gen

**类型**：`quiz_gen`  
**作用**：生成题目、选项或答案解析。

适用场景：

- 自测
- 章节练习
- 题库构建

#### ⚖️ 对比辨析节点 · Compare

**类型**：`compare`  
**作用**：对多个概念、方案或资料进行对照分析。

适用场景：

- 模型或理论比较
- 资料差异分析
- 观点对照

#### 🧠 思维导图节点 · Mind Map

**类型**：`mind_map`  
**作用**：将知识结构整理成导图式层级文本。

适用场景：

- 快速展示知识全貌
- 用于复盘和二次展示
- 从总结结果转为结构图谱

#### ✏️ 合并润色节点 · Merge Polish

**类型**：`merge_polish`  
**作用**：把多个来源的内容整合为统一风格结果。

适用场景：

- 合并不同节点产出
- 最终文稿润色
- 输出前的语言和结构统一

### 4.5 交互类节点 (Interaction)

#### 💬 智能对话节点 · Chat Response

**类型**：`chat_response`  
**作用**：把工作流结果转化为面向用户的回答、解释或引导。

适用场景：

- 对用户进行结果反馈
- 输出“下一步建议”
- 作为工作流与聊天交互之间的桥梁

### 4.6 输出类节点 (Output)

#### 💾 数据持久化节点 · Write DB

**类型**：`write_db`  
**作用**：将工作流结果持久化保存。

适用场景：

- 保存最终结果
- 沉淀结构化数据
- 为后续统计或复用提供基础

#### 📥 文件导出节点 · Export File

**类型**：`export_file`  
**作用**：把内容转为文件形式供下载或分发。

适用场景：

- 导出总结结果
- 导出题库、文稿或学习材料
- 形成可分享成果

### 4.7 结构类节点 (Structure)

#### 🗂️ 循环容器节点 · Loop Group

**类型**：`loop_group`  
**作用**：作为循环区域的结构容器，承载内部循环节点。

适用场景：

- 可视化表达循环块
- 批量处理链路的结构化包裹
- 让复杂工作流在画布上更清晰可维护

说明：

- `loop_group` 是当前真实节点体系的一部分
- 它属于结构节点，不等同于普通执行节点

## 第五章 · 典型工作流示例

### 5.1 深度学习型

适合从一个复杂主题出发，生成一条完整学习路径。

示例链路：

```text
trigger_input
-> ai_analyzer
-> ai_planner
-> outline_gen
-> content_extract
-> summary
-> chat_response
```

适合场景：

- 某门课程入门
- 某篇论文拆解
- 某专题知识整理

### 5.2 练测一体型

适合把学习内容直接转化为复习材料和测验题。

示例链路：

```text
trigger_input
-> knowledge_base
-> content_extract
-> flashcard
-> quiz_gen
-> export_file
```

适合场景：

- 考试复习
- 闪卡制作
- 自测训练

### 5.3 联网研究型

适合需要外部公开资料参与的学习或研究任务。

示例链路：

```text
trigger_input
-> web_search
-> ai_analyzer
-> compare
-> summary
-> chat_response
```

适合场景：

- 新领域快速调研
- 比较不同观点和资料来源
- 对公开话题做研究摘要

### 5.4 知识库深化型

适合围绕已有资料做深度结构化整理。

示例链路：

```text
trigger_input
-> knowledge_base
-> outline_gen
-> mind_map
-> merge_polish
-> write_db
```

适合场景：

- 课程资料系统化
- 私有知识沉淀
- 长期知识库整理

### 5.5 条件分流与循环处理型

适合材料量大、需要按批次或按条件拆分处理的场景。

示例链路：

```text
trigger_input
-> ai_analyzer
-> logic_switch
-> loop_group
-> loop_map
-> summary
-> export_file
```

适合场景：

- 批量处理章节
- 对不同输入走不同路径
- 对一组材料重复执行同一逻辑

## 第六章 · 快速上手指南

### Step 1 · 明确目标

先把你真正要解决的问题说清楚。  
比起“帮我学这个”，更推荐这样的输入：

- 我想把这份资料整理成考试复习手册
- 我想把这个主题拆成一周学习计划
- 我想把论文内容变成大纲、总结和题库

目标越明确，系统越容易生成清晰的工作流骨架。

### Step 2 · 生成与整理工作流

进入前端后，你可以：

- 让系统生成工作流
- 自己补充节点
- 调整顺序与布局
- 删除不需要的步骤

本地开发默认端口：

- 前端：`http://localhost:2037`
- 后端文档：`http://localhost:2038/docs`

### Step 3 · 运行并观察流式结果

执行工作流后，可以观察：

- 节点状态变化
- 节点输入快照
- 流式 token 输出
- 完成或报错状态
- 循环迭代过程

这也是 StudySolo 和纯聊天工具最大的差异之一：你看到的是过程，而不是只有结论。

### Step 4 · 保存、复用与分享

当工作流可用后，你可以继续把它当作资产维护：

- 保存为自己的工作流
- 再次运行
- 收藏、点赞、公开
- 在协作场景下继续扩展

如果你的目标是长期复用某类学习链路，最佳做法不是重复写提示词，而是沉淀一条稳定工作流。

## 结语

StudySolo 试图把学习这件事从“零散提问”升级为“过程可设计、结果可复用、执行可观察”的工作流体验。

如果你需要的是一个只会即时回答问题的聊天框，那么它看起来也许有些“重”；  
但如果你希望把长期学习、知识整理和内容产出做成一套稳定方法，那么 StudySolo 更适合被理解为：

> **一张可执行的学习蓝图。**
