## 答辩前必读 · 备查区（不念，追问时翻阅）

### 仓库与版本

- **Polyglot Monorepo**：`frontend/`（Next.js 16，端口 2037）、`backend/`（FastAPI，2038）、`agents/`（多微服务 8001–8099）、`introduce/`（Vite 介绍页）、`shared/`（Git Submodule）、`supabase/migrations/`。
- **竞赛提交版**冻结于 GitHub `AIMFllys/StudySolo`；**本仓库**为活跃开发分支。

### 前端核心路径

- 画布：`@xyflow/react` + Zustand（`use-workflow-store`），Undo/Redo、执行追踪 UI。
- 画布 → AI：`use-canvas-context.ts` 的 `serialize()` 生成 `CanvasContext`（节点摘要、DAG 文本、选中节点、坐标），由 `SidebarAIPanel` → `useStreamChat` 发送。**不含**全站路由自动感知。
- 三种模式：Chat / Plan / Create；Create 下区分 BUILD（空画布 → `/api/ai/generate-workflow`）与 MODIFY（增量 JSON + 重试）。
- 统一流式入口：`/api/ai/chat`，实现于 `backend/app/api/ai/chat.py`。

### 后端核心路径

- DAG 引擎：`backend/app/engine/` 模块化架构 → `executor.py` / `topology.py` / `node_runner.py` / `level_runner.py` / `loop_runner.py`。
- SSE 事件：`node_input` → `node_status` → `node_token` → `node_done` → `loop_iteration` → `workflow_done`。
- 执行 API：`POST /workflow/{id}/execute`（`backend/app/api/workflow/execute.py`）。
- 节点注册：`backend/app/nodes/__init__.py` 自动发现 → `NODE_REGISTRY`。
- Prompt 体系：`backend/app/prompts/` 模块化 Markdown + 模板变量。

### 节点口径

- 参赛说明书写 **18 种**（冻结版）；当前代码枚举 **20 项**（含 `community_node`、`loop_group`）。口播说 **「20 种节点类型」** 或 **「18 种学习向 + 控制流与社区扩展」** 均可。

## P1 · 开场：一个场景，点燃共鸣（约 45s）

**【PPT 提示】**：一张「期末两周倒计时」的场景插画或截图；旁列六个痛点关键词，不要编号。

**口播：**

各位评委老师好，我们是 StudySolo。

先讲一个常见场景：**期末还剩两周**，你在聊天工具、笔记软件、题库网站之间来回切换。问题不是 AI 不聪明，而是流程断裂：前一步结果要手动复制到下一步，过程不可见，结果难复用。我们调研后把它归纳为六类痛点：路径不清、工具割裂、过程黑盒、成果沉淀困难、提问门槛高、效果难评估。

StudySolo 的核心就是把这条碎片链路，变成一条**可编排、可执行、可追踪、可复用**的学习工作流。  
我们已经形成一整套落地能力：自然语言选题生成工作流、执行面板可观测、学习产物可渲染、知识库沉淀与导出、子后端 Agent 专业能力接入、以及 SOP 驱动团队协作。

## P2 · 我们做什么 + 凭什么是我们（约 55s）

**【PPT 提示】**：上半页一句话定位；下半页三列竞品对比表（ChatGPT、Dify/Coze、StudySolo）。

**口播：**

一句话定位：**用户用自然语言描述学习选题，系统在画布生成并迭代学习工作流，按依赖自动执行，过程全程可见，成果持续沉淀复用**。

和通用聊天工具相比，我们把单轮问答升级成多步可编排流程；  
和通用工作流工具相比，我们有学习场景专属节点；  
和手工拼工具相比，我们在一张画布里完成检索、分析、生成、测验和导出。  
更重要的是，平台 AI 不只用于建图，也支持持续对话与学习打卡；工作流可保存、可追踪、可 Fork 协作，学习过程本身成为资产。

## P3 · 架构：三层讲清楚（约 70s）

**【PPT 提示】**：一张分层架构图（意图层 → 编排层 → 执行与路由层），右侧小字列技术栈。

**口播：**

整个系统可以概括为三层。

**第一层，意图理解与交互层**。前端用可视化画布承载学习流程；每次对话都会把当前画布结构序列化给后端，后端先做意图分类，再决定是聊天、规划还是直接改图，空画布可一键生成整图。

**第二层，DAG 编排与执行引擎**。我们自研拓扑排序与分层调度，支持循环与分支。节点执行过程中通过 SSE 持续回推状态、Token 和结果，用户可以看到每一步发生了什么。

**第三层，多模型路由与治理层**。后端对接多平台模型并支持容灾降级；数据层基于 Supabase + RLS + JWT；同时有运行记录、审计日志和管理后台，支撑可治理的长期运营。

## P4 · 演示：最短闭环（约 55s）

**【PPT 提示】**：录屏截帧或现场操作：工作区 → 输入 → 画布生成 → 运行 → 执行追踪。

**口播：**

演示走一条最短闭环。  
先在工作台输入「帮我做期末复习流」，系统自动生成学习工作流并连线。  
再用一句话增量修改，比如「把闪卡放到测验后面」，画布立即更新并支持撤销。  
点击运行后，你会看到两层可视化：节点下方实时滚动流式内容，右侧执行面板实时显示整条流程跑到哪一步。  
执行完成后，可立刻查看运行日志；题目、闪卡、导图、报告等结果可在画布直接交互，也可一键沉淀到知识库或导出文件。  
如果流程里有同层无依赖节点，系统会并发执行，整体效率会明显提升。

## P5 · 创新点与壁垒（约 45s）

**【PPT 提示】**：四个图标 + 底部一句「飞轮」。

**口播：**

我们的壁垒不是“功能多”，而是“学习场景做得深”：  
第一，我们是学习垂直工作流，不是通用聊天。用户从学习选题出发，系统直接通过工作流 API 生成、执行、自测、沉淀，形成完整闭环。  
第二，平台支持 MCP 与 Skills 协同，可指导 AI 远程调用工具能力，更自然融入真实学习任务。  
第三，部分节点本身就是子智能体节点，深度集成在工作流里，通过 Agent Gateway 接入专业能力，不是外挂插件。  
第四，执行过程可视化很具体：节点卡片有实时流式输出，执行面板有全流程状态，执行后还能追运行日志。  
第五，结果不是一段文字，而是可直接使用的学习成果：闪卡、测验、导图、报告，并且可入库、可导出、可复用。

这些能力形成飞轮：流程越可复用，协作越高效；协作越高效，平台能力增长越快。

## P7 · 团队与收尾（约 40s）

**【PPT 提示】**：三人头像 + 职责关键词 + 二维码/URL。

**口播：**

团队三人。**羽升**负责核心架构、执行引擎、AI 路由和部署；**小李**负责子 Agent 与 Gateway 接入；**小陈**负责用户文档、Wiki 与调研闭环。  
我们不仅有分工，还有协作机制：通过 Agent 分支提交 SOP、README 交接和审核接力，把多人协作做成稳定流水线。

最后一句：**StudySolo 不是又一个 AI 聊天框，而是一个让学习流程可创建、可执行、可复用、可协作进化的平台**。
