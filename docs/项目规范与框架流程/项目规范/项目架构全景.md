## 1. 项目定位与核心流转

StudySolo 是 1037Solo 生态中的 **AI 学习工作流引擎与知识计算平台**。

**核心生命周期流转：**
1. **画布编排 (Frontend Canvas)**: 用户在 `/c/[id]`（私有画布）或 `/s/[id]`（公开视图）中组装具有不同输入、分析、生成的节点。
2. **DAG 序列化 (Payload)**: 前端提取节点（nodes）与有向边（edges），转换为 JSON 结构并构建执行图。
3. **引擎调度 (Backend Engine)**: FastAPI 接收后通过拓扑排序确定执行顺序，将任务分发至各自的 Node Handler。
4. **模型路由 (AI Router)**: 基于 `ai_model_skus` 表和 `config.yaml` 定义的路由策略（如 `native_first` 或 `proxy_first`），自动分配底层 LLM 厂商并执行请求。
5. **实时回流 (SSE Stream)**: 引擎利用 SSE 向前端实时推送节点状态、增量 Token 及循环迭代进度（`node_status` / `node_token` / `node_done` 等）。
6. **数字资产持久化 (Data Layer)**: 输出结果根据节点逻辑，可被写入系统知识库（Supabase / pgvector）、触发文件导出下载，并将 AI API 耗时/消耗记录计入用量账本（`ss_ai_usage_ledger` / `cost_amount_cny`）。

## 4. 前端架构拓扑

前端体系遵循 **领域主导设计 (DDD 切片)** 及 **"App Router + Client Component"** 平衡模式。

### 4.1 页面路由基石 (`src/app/`)
*   `/(dashboard)/workspace`: 控制台入口，工作空间面板（由 `Navbar` 和 `MobileNav` 限定在 `DashboardShell` 下）。
*   `/(dashboard)/c/[id]`: 私有画布编辑器，要求拥有写或读权限的完整验证路由。
*   `/s/[id]`: 公开脱敏分享视图，适配全尺寸访客访问展示。
*   `/(admin)/`: 后台系统，采用严密的独立 Layout 包裹。

### 4.2 垂直业务领域 (`src/features/`)
被高度封装的业务上下文聚集点：
*   **`workflow/`**:
    *   `canvas/`: `WorkflowCanvas` 实现、微型地图 (`CanvasMiniMap`)、拖拽缩放封装。
    *   `nodes/`: 每种节点的特定 React UI 容器（如带有动效的 `AnimatedEdge`，或 `AIStepNode`, `LoopGroupNode`）。
    *   `panel/` / `toolbar/`: 节点侧边栏检查器及全局执行命令区。
    *   `hooks/`: 与 Zustand 深度绑定的画布驱动（如 `use-action-executor`, `use-stream-chat`）。
*   **`admin/`**: 分模块涵盖大盘（dashboard），审计（audit），通知（notices），计费策略（config），及人员控制（users, members）。
*   **`knowledge/`**, **`auth/`**, **`settings/`**: 其他相对独立的纯业务域隔离。

### 4.3 全局权限拦截 (`middleware.ts`)
保护路由策略已硬编码：
1. 提取请求域的 `access_token` Cookie。
2. 对 `/c/` 及其子路径、`/workspace` 执行强拦截，重定向回 `/login` 带入来源 `redirect` 标志。
3. 公开路由（`/`, `/s/` 等）静默放行。

## 5. 后端架构解剖

后端采用了**强逻辑分层原则**与**显式路由合并模式**进行统一调度。

### 5.1 Restful API 入口 (`backend/app/api/router.py`)
所有的业务接口均按照域和职责以 `include_router` 显式分类，并通过前缀进行访问网关切分。

**工作流主控 (Workflow & Execute):**
*   `/workflow` (`workflow_router`): 节点的 CRUD、创建、查找列表。
*   `/workflow/execute` (`workflow_execute_router`): 执行触发器。
*   `/workflow/social` (`workflow_social_router`): 点赞、收藏、公共推荐列表及浏览数。
*   `/workflow/collaboration` (`workflow_collaboration_router`): 多人只读/编辑共建者授权与移除管理。

**知识模型与计算 (AI, Nodes & Documents):**
*   `/ai/generate-workflow`, `/ai/chat`, `/ai/chat-stream` (`ai_router` / `ai_chat_router`).
*   `/ai/models/catalog` (`ai_catalog_router`): 基于 SKU 主键的动态用户侧模型查询端面。
*   `/nodes`: 读取工作流可用节点 Schema 和描述规则。
*   `/knowledge`, `/exports`: 文档切片读取和最终产物打包通道。

**平台系统审计 (Admin & Usage & Feedback):**
*   `/usage` (`usage_router` / `services/usage_ledger.py`): Live 时序打点，总花费分摊看板，Tokens 账单计算。
*   `/admin/*` (`admin_dashboard`, `admin_models`, `admin_audit` 等): 管理中心各项数据源。

### 5.2 核心中间件安防 (`app/middleware/`)
1.  **`SecurityHeadersMiddleware` / `CORSMiddleware`**: 控制头伪造和防止滥用。
2.  **`JWTAuthMiddleware`**: `auth.py` 中截取 Bearer Token 解析 `supabase.auth.user()` 作为全局身份凭证。
3.  **`AdminJWTMiddleware`**: 基于 JWT 附加的声明或特殊查询单点拦截所有管理级端点访问。

## 6. 工作流引擎体系深潜

在 `backend/app/engine/` 和 `frontend/src/types/workflow.ts` 形成闭环的强逻辑约束系统。

### 6.1 节点形态定义
系统共支持三大类共计 19 种标准化节点（事实节点表）：

**🟢 输入层节点 (3种)**
*   `trigger_input`: 基础启动引流触发槽。
*   `knowledge_base`: 从挂载的数据源 (`pgvector`) 进行余弦相似或关键字混合检索召回。
*   `web_search`: 打通公共互联网外挂搜索库，供给结构化事实资料。

**🔵 分析与引擎节点 (6种)**
*   `ai_analyzer`: 输入意图研判与字段推断。
*   `ai_planner`: 对高模复合任务分段计划构建。
*   `compare`: 双视角异同评判逻辑矩阵对比。
*   `logic_switch` **(P2引擎)**: 根据数据状态的条件分支守卫拨片。
*   `loop_map` **(P2引擎)**: 针对数组类对象的映射执行分化槽。
*   `loop_group` **(结构/容器类)**: 前端虚拟渲染成一块框体容器的循环聚合层。

**🟣 生产输出与交付节点 (10种)**
*   `outline_gen`, `content_extract`, `summary`: 文本萃取、结构重组系内容生产。
*   `flashcard`, `quiz_gen`, `mind_map`: 记忆强化、检测及关系脑图知识产物。
*   `merge_polish`: 语料终态组装校正池。
*   `chat_response`: 同用户完成阻塞型会话指令互动。
*   `export_file`, `write_db`: 本地文件系统输出器与持久化表写入器。

### 6.2 线路与生命周期
*   **Edge 类型**: 目前引擎约定所有的物理折线在渲染层及序列化中只有 **`sequential`** (顺序承接)。如果伴随流转状态，参数储存于 `data.branch`（分支挂载标志位）。
*   **Node Status 状态流**: `pending`(未开始) -> `running`(运转) | `waiting`(挂起或睡眠等待) -> `done`(成功) / `error`(异常) / `skipped`(跳过执行) / `paused`(断点暂停)。

### 6.3 服务器下推事件流 (SSE Payload Events)
引擎在工作时，会不间断抛出下述 7 个主事件供下游解析还原画面：
1.  `node_status` (更新渲染外壳样式)
2.  `node_input` (节点接收到并序列化输入镜像内容)
3.  `node_token` (AI 吐字微事件，推给 markdown renderer)
4.  `node_done` (节点完整执行结束伴随完整聚合响应)
5.  `loop_iteration` (返回外包结构执行帧数反馈)
6.  `workflow_done` (执行组生命周期消亡)
7.  `save_error` (持久化过程错误中断事件)

## 7. AI 模型路由矩阵与新计费标准

平台引入了全新的企业级 AI 模型分层管理策略。

### 7.1 分层路由哲学 (`backend/config.yaml`)
*   **Providers (平台供应)**: 明确为 `dashscope` (阿里云百炼), `deepseek`, `moonshot`, 加入 `qiniu`, `compshare` (作为分发商角色)。区分 `role` (native 直通 / proxy 代理服务)。
*   **Task Routes (挂载面)**:
    每个模块定义执行特权 (`routing_policy`)。例如 `routing_policy: "native_first"` 优先确保生成节点 (`summary`, `content_extract` 等) 分配至 `sku_dashscope_qwen_plus_native` 或 `sku_deepseek_reasoner_native` 等。

### 7.2 模型目录表 (`Supabase: ai_model_skus` & `ai_model_families`)
*不再以散装硬编码为准，现在具有计费基准强制表:*
*   **核心字段溯源**:
    *   `sku_id` (主键，如 `sku_qiniu_kimi_k2_5_proxy`)
    *   `billing_channel` (`native`, `proxy`, `tool_service`)
    *   `price_source` / `pricing_verified_at` (价格锚定源头与效时)。

### 7.3 全局计费与用法沉淀基座 (`usage_ledger`)
*   一切对 LLM API 以及具有消耗特性的调用，统一记录于 `ss_ai_usage_ledger` 和 `ss_ai_usage_minute` (窗口聚合表)。
*   计费口径严禁使用历史的 USD，而是以人民币百万 Token 的微分为主导：
    *   `input_price_cny_per_million`
    *   `output_price_cny_per_million`
    *   `cost_amount_cny` / `total_cost_cny`。
*   请求追溯由 `logical_requests` 串联起一系列实际对外发出的 `provider_calls`。

## 8. 数据库设计基线与多域隔离

架构基于 Supabase PostgreSQL 进行 RLS （行级安全）细粒度权控。

### 8.1 域隔离前缀
平台采取表前缀切面以规避项目名称冲撞和界定同步方向：
*   **(无级别前缀)**: 为跨站点生态（如包含 1037Solo 主平台及外部论坛）**高度共享同步池** (Shared Context)。
*   **`ss_`**: 为 **StudySolo 的排他域** (Exclusive Context)。
*   **`pt_`**: 代表主 Platform 承销层 (如支付台业务)。

### 8.2 数据表分布快照
**全局 Shared 表 (核心资产池):**
*   `user_profiles`, `subscriptions`, `addon_purchases`, `payment_records`
*   `ai_model_families`, `ai_model_skus` (新全局计费表)
*   `verification_codes_v2`, `auth_rate_limit_events` (授权控制与验证码安全层)

**StudySolo 核心表 (功能域池):**
*   **执行引擎组**: `ss_workflows`, `ss_workflow_runs`
*   **AI使用度量**: `ss_ai_conversations`, `ss_ai_messages`, `ss_ai_usage_events`, `ss_usage_daily`
*   **系统运营组**: `ss_notices`, `ss_ratings`, `ss_feedback`, `ss_system_config`
*   **知识库与向量**: `ss_kb_documents`, `ss_kb_document_chunks`, `ss_kb_chunk_embeddings` (`pgvector` 高维投射数据)

## 9. 源代码共享机制 (Shared Layer)

**绝对规范声明：**
在当前 StudySolo Next.js + FastAPI 架构内，`shared/` 是作为 **Git Submodule** (事实源读取自 `.gitmodules`) 将一套 Types/Schemas 共享至本仓库。
而在远端的 Platform 总体宏观库视角中，`StudySolo/` 项目本身充当的是 **Git Subtree** 桥梁结构。这两层含义在文档操作与 PR 合并过程中绝对不可被张冠李戴（参见 `shared/docs/guides/subtree-sync.md`）。

## 10. 全栈联调执行流

在修改与排错时，保持对应端口的环境清洁以防止 `EADDRINUSE` 错误发生。

**节点服务启停指南:**
*   **Web / Dashboard 端 (Frontend)**:
    端口占用 `2037`。通过 `cd frontend && pnpm dev` 使用 Turbopack 热更新构建器。
*   **ASGI Rest 服务端 (Backend)**:
    端口占用 `2038`。通过 `cd backend && uvicorn app.main:app --reload --port 2038` 装载挂载应用层脚本。
*   **聚合拉起方式**:
    利用 `scripts/start-studysolo.ps1` 唤醒全局工作台服务。

## 11. 文档不可变审计与迭代公约

为防止 AI Agent 或维护者信息漂移，架构白皮书要求：
1. **源码决断**: `package.json` 的版本、`types/workflow.ts` 的结构定义与接口参数模型（Router）为至高权威真理。历史的架构构想草案（如带有旧名字段或未落地概念）不再在此文档留存。
2. **强制同步项**: 当下述任何组件进行重构后，此篇 《项目架构全景》 **必须被主动覆盖连带更新**：
    *   API 路由命名空间增删。
    *   Workflow Node / Edge / 引擎生命周期 Status 与 Payload 事件更迭。
    *   Supabase 新增表格或字段，乃至 AI Sku 定价的汇率底包和模型名称换绑。
    *   `.gitsubmodule` 仓库同步流方向重设。
3. **格式基线**: 中文文档严格使用 **UTF-8（非 BOM 指纹）及 LF 换行** 以匹配 Bash Shell Script 安全审计需求。
