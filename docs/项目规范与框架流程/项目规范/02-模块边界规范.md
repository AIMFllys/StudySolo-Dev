# 模块边界规范

> 文档版本：1.0.0
> 创建时间：2026-04-09
> 基于：docs/plan/TeamNewRefactor/ 系列分析文档

---

## 1. 概述

本文档定义 StudySolo 项目中各模块之间的边界、接口规范和依赖规则，确保团队协作时减少冲突、提高代码可维护性。

---

## 2. 前端模块边界

### 2.1 Feature Module 边界

**规则：**
1. Feature 模块是独立开发单元，有明确的 owner
2. Feature 之间通过 public API 通信，不直接导入内部组件
3. 共享组件必须移到 `packages/ui/` 或 `components/ui/`

**当前 Feature 模块：**

| 模块 | 路径 | Owner | 职责 |
|------|------|-------|------|
| workflow | `features/workflow/` | 主开发者 | 工作流编辑器核心 |
| admin | `features/admin/` | 主开发者 | 管理后台 |
| auth | `features/auth/` | 主开发者 | 认证相关 |
| knowledge | `features/knowledge/` | 待定 | 知识库（考虑合并）|
| community-nodes | `features/community-nodes/` | 待定 | 社区节点 |
| settings | `features/settings/` | 待定 | 用户设置 |

### 2.2 Store 边界

**规则：**
1. Store 只管理状态，不直接执行网络请求
2. Store 之间不允许隐式耦合（不能在一个 store 里直接调用另一个 store 的 state）
3. 跨 store 同步必须在调用方（hook 或组件）显式处理

**当前 Store：**

| Store | 路径 | 职责 |
|-------|------|------|
| use-workflow-store | `stores/` | 工作流状态 + 执行 + 历史 |
| use-ai-chat-store | `stores/` | AI 聊天 UI 状态 |
| use-conversation-store | `stores/` | 对话历史持久化 |
| use-panel-store | `stores/` | 面板开关状态 |
| use-settings-store | `stores/` | 主题和偏好 |
| use-admin-store | `stores/` | Admin sidebar |

**跨 store 耦合问题（待修复）：**
- `useAIChatStore.pushMessage` 直接调用 `useConversationStore.getState()`
- 修复方案：在 `use-stream-chat.ts` 中显式处理同步

### 2.3 Service 层边界

**规则：**
1. 所有 API 调用必须通过 `services/` 层
2. 使用 `api-client.ts` 提供的 `authedFetch` 而非裸 `fetch`
3. Service 之间不能循环依赖

**当前 Service：**

| Service | 路径 | 职责 |
|---------|------|------|
| api-client | `services/` | 统一 fetch + 认证 + 错误处理 |
| workflow.service | `services/` | 工作流 CRUD |
| auth.service | `services/` | 认证 API |
| admin.service | `services/` | Admin API |

**问题（待修复）：**
- `workflow.server.service.ts` 重复实现了 `workflow.service.ts` 的部分函数

## 3. 后端模块边界

### 3.1 API 路由边界

**规则：**
1. API 路由只做入参校验 + 权限控制，不承载长业务逻辑
2. 业务逻辑下沉到 `services/` 层
3. 按 domain 分组，避免碎片化

**当前问题（待修复）：**

| 问题 | 文件 | 建议 |
|------|------|------|
| workflow 路由碎片化 | 5 个文件在 `/workflow` 前缀 | 合并到 `api/workflow/` 子目录 |
| /ai 前缀碎片化 | 5 个文件在 `/ai` 前缀 | ai_chat + ai_chat_stream 合并 |
| admin 路由分散 | 9 个文件在 `/admin` 前缀 | 可接受，或按数据/配置分组 |

### 3.2 Service 层边界

**规则：**
1. Service 是独立业务逻辑单元
2. Service 只能调用下层（database, external API），不能调用 API 层
3. 共享逻辑提取到公共 Service

**当前 Service：**

| Service | 路径 | 职责 |
|---------|------|------|
| ai_router | `services/` | AI 多平台路由（边界模糊，待修复）|
| llm_caller | `services/` | HTTP 调用（应只做调用）|
| llm_provider | `services/` | provider client 管理 |
| usage_ledger | `services/` | 使用量记录（交叉 concern，待优化）|
| knowledge_service | `services/` | 知识库 facade |
| file_parser | `services/` | 文件解析 |

**问题（待修复）：**

| 问题 | 建议 |
|------|------|
| ai_router 包含流式逻辑 | 拆分：router 只做路由，流式提取到 generators/ |
| usage_ledger 模式在多个 API 文件重复 | 提取为 `@track_usage` 装饰器 |

### 3.3 节点模块边界

**规则：**
1. 节点是独立插件，继承 `BaseNode`
2. 节点只能通过 `BaseNode` 定义的接口与引擎交互
3. 节点配置通过 `node_config` 传递，不硬编码

**当前节点结构：**
```
nodes/
├── _base.py              # 基类
├── nodes/                # 官方节点
│   ├── input/
│   ├── analysis/
│   ├── generation/
│   └── ...
└── plugins/             # 社区插件（待建立）
```

## 4. 模块间通信协议

### 4.1 前端 → 后端

**协议：REST API + JSON**

```typescript
// 请求
fetch('/api/workflow', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`,
  },
  body: JSON.stringify(data),
});

// 响应
{
  "data": {...},
  "error"?: string,
}
```

### 4.2 后端 → 子后端 Agent

**协议：OpenAI 兼容 HTTP API**

```
POST /v1/chat/completions
Content-Type: application/json

{
  "model": "agent-name",
  "messages": [...],
  "stream": false
}
```

**响应：**
```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "choices": [{
    "message": {"role": "assistant", "content": "..."}
  }]
}
```

## 5. 依赖规则

### 5.1 前端依赖方向

```
components/ui/  ←  所有模块可依赖
stores/         ←  components/ 和 hooks/ 可依赖
hooks/          ←  components/ 可依赖
features/       ←  不能相互依赖（除非通过 public API）
services/       ←  features/ 可依赖
types/          ←  所有模块可依赖（只读）
```

### 5.2 后端依赖方向

```
models/         ←  所有层可依赖（只读数据模型）
core/           ←  api/, services/, engine/ 可依赖
services/       ←  api/ 可依赖
api/            ←  不能依赖 engine/（防止循环）
engine/         ←  nodes/ 可依赖
nodes/          ←  不能依赖 api/, services/
```

## 6. 新增模块流程

### 6.1 前端新增 Feature

1. 在 `features/` 下创建目录
2. 创建 `index.ts` 导出 public API
3. 在 `features/*/index.ts` 中声明 public 接口
4. 更新本文档的模块表

### 6.2 后端新增 Service

1. 在 `services/` 下创建文件
2. 在 `services/__init__.py` 导出（如果需要）
3. 在 `api/` 相应路由中使用
4. 更新本文档的模块表

### 6.3 新增节点

1. 在 `nodes/nodes/<category>/` 下创建
2. 继承 `BaseNode`，设置 `node_type`
3. 注册到 `NODE_REGISTRY`
4. 前端在 `RENDERER_REGISTRY` 添加对应渲染器

---

## 7. 违规处理

| 违规类型 | 处理方式 |
|---------|---------|
| Feature 间直接导入 | Code review 拒绝，要求通过 public API |
| Store 隐式耦合 | 标记为 tech debt，限期修复 |
| API 层调用 Service 外的逻辑 | Code review 拒绝 |
| 循环依赖 | CI 失败，必须解决 |
