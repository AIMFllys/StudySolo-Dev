# Pull Request 与 Code Review 规范

> 最后更新：2026-04-09

## 1. Pull Request 规范

### 1.1 PR 标题

与 Commit 规范一致：`<type>(<scope>): <subject>`

### 1.2 PR 描述模板

```markdown
## 描述
<!-- 简要说明这个 PR 做什么 -->

## 类型
- [ ] 新功能 (feat)
- [ ] Bug 修复 (fix)
- [ ] 重构 (refactor)
- [ ] 文档 (docs)
- [ ] 安全修复 (security)
- [ ] 其他

## 影响的模块
- `backend/app/api/`
- `frontend/src/features/`

## 测试
- [ ] 单元测试通过
- [ ] 手动测试通过
- [ ] 不需要测试

## 检查清单
- [ ] 代码遵循项目规范
- [ ] 没有 console.log / debugger / print 调试
- [ ] 无硬编码凭证或密钥
- [ ] 相关注释/文档已更新
```

### 1.3 合并策略

| 场景 | 合并方式 |
|------|----------|
| Feature 分支 → main | **Squash and Merge**（推荐，保持 main 历史简洁） |
| 需要保留完整历史的重构 | Merge Commit |
| 紧急安全修复 | Squash and Merge，加速合并 |

### 1.4 模块边界原则

> **一个 PR 尽量只属于一个模块**

- Wiki PR **不碰**主工作流逻辑
- 子后端 PR **不碰**核心执行引擎
- 如确实需要跨模块，在 PR 描述中明确说明原因

---

## 2. Code Review 检查清单

### 2.1 功能审查
- [ ] 功能按需求实现
- [ ] 边界情况已处理
- [ ] 错误处理合理

### 2.2 代码质量
- [ ] 代码简洁易读
- [ ] 函数长度合理（< 100 行）
- [ ] 无重复代码
- [ ] 命名清晰有意义

### 2.3 安全
- [ ] 无硬编码凭证
- [ ] 输入已验证
- [ ] SQL 注入已防护
- [ ] XSS 已防护
- [ ] RLS 策略正确

### 2.4 测试
- [ ] 覆盖关键路径
- [ ] 测试可重复执行

### 2.5 Review 注释规范

```text
👍  确认通过
💡  建议：可以考虑...
❌  问题：这个实现存在...
🔴  阻塞：此改动需要讨论后才能合并
```
