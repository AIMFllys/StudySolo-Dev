# 工作流节点系统后续演进与技术遗留全景分析

> **编写时间**：2026-03-27
> **位置**：`docs/Plans/daily_plan/workflow_canvas/`
> **目标**：在完成所有的“体系注册、架构纠偏、元数据底座打通”之后，极为精确地梳理**系统距离真正的“生产级完备”与“代码级整洁”还差什么**。本文档也是下一阶段（或下一个 Sprint）的发版提纲。

---

## 摘要 (Executive Summary)

经过此前的多轮次爆改（AI 提示词对齐、`BaseNode` 体系完善、执行追踪抽屉贯通、Manifest 动态表单化），工作流系统的**骨架已经100%硬化**。目前 19 个节点均已注册，没有任何链路是直接断裂断的。

但**骨架长成，不代表器官足够强大**。目前剩下的工作完全属于“三大深水区”：
1. **执行引擎（Executor）的深度分发控制不足**（典型表现：`loop_map` 无法真实迭代子图）。
2. **重点节点的终端业务闭环度低**（典型表现：只负责生成 JSON，不去管用户拿到 JSON 之后怎么学习、怎么导出）。
3. **沉重的全局级技术架构负担**（巨石组件与 Lint 历史债）。

以下是极为详尽的各模块断点还原与待解决任务池：

---

## 一、 执行引擎层深度打磨（核心架构级）

这是直接制约复杂业务组装能力的硬核问题。

### 1. `loop_map` 节点真正的迭代分发缺失
- **当前现状**：`loop_map` （内容拆分/映射节点）目前的作用仅停留在**后端字符串切分**。它能利用 LLM 拆出 `[{item: "...", label: "..."}]` 的 JSON 数据并传递下去。但目前的执行引擎（`executor.py`）拿到这组数据后，只会简单把它当做一个长字符串或者单节点结果传递给直接下游节点。
- **还差什么**：
  - `executor.py` 必须实现**上游 Array 数据拦截机制**。
  - 识别上游节点的 `is_iterable: True` 属性。
  - 为数组中的**每一个元素，单独触发下游整个局部子图**（迭代/Map 操作）。
  - 需要复杂的并发分发收集汇总（Scatter-Gather），目前引擎代码中 `get_branch_filtered_downstream` 虽然完成了逻辑分支跳转，但还没写真正的 Array 遍历克隆逻辑。

### 2. 全局状态图与终止态感知
- **当前现状**：执行引擎是基于广度/深度（Graph-walk）执行的，且使用了事件推流（SSE）。
- **还差什么**：
  - 画布上缺少明确的**工作流进度条计算策略**（例如：总节点 10 个，目前处于 2/10），目前前端无法感知“还剩多长时间”。
  - 执行链追踪面板（Trace Drawer）在节点崩溃或图中有环的情况下，其错误恢复能力比较弱。

---

## 二、 节点业务闭环深度扩展（前端与交互层）

这是决定产品“好不好用、能不能卖”的业务价值断点。

### 1. 知识库节点（Knowledge Base）的 RAG 精细化溯源
- **当前现状**：动态配置抽屉已支持基于 `document_ids` 进行检索并传递内容。
- **还差什么**：
  - **引用段落高亮（Citation / Source Tracking）**：当后续 LLM 节点根据知识库吐出内容时，工作流界面完全无法像传统 RAG 产品那样，给回答标上 `[1]` 的脚标并在执行面板直接点击滑出原文档。
  - **动态参数状态留存**：检索时的 chunkSize、Overlap 等复杂高级索引构建参数，暂无专门的设置界面。

### 2. Flashcard（闪卡节点）的学习链路流转
- **当前现状**：只会机械地生成正面/反面结构的 JSON，前端能翻转。
- **还差什么**：
  - **Anki / CSV 物理包导出**：这是闪卡最刚需的点，必须有 `.apkg` 或 `.csv` 的下载生成后端逻辑。
  - **简便进度机制**：没有集成类似 SM2 的间隔重复概念。这让 Flashcard 更像是占位 UI，而不是学习工具。

### 3. Quiz_gen（测验生成）的数据沉淀
- **当前现状**：提供四选一 json 渲染，能乱序，能点击看答案。
- **还差什么**：
  - 点击“提交”或产生分数后，目前的状态只活在这个 React 渲染器里。缺少将答题结果**写回个人统计数据库**，或者将做错的题目推送给下游节点的逻辑连接词。

### 4. 数据库存储节点（Write DB）的高级冲突策略
- **当前现状**：将 LLM 内容挂载到一个指定的 JSON Key（存给 User Profile 或者当前 Run Session 当中）。
- **还差什么**：如果写入目标的 Key 已经有值了，目前没有做 `Append (追加)` vs `Override (覆盖)` vs `Abort (报错提示)` 的细粒度交互处理。

---

## 三、 表单与配置框架的交互成熟度（UX/UI层）

### 1. 托拽连线时的“默认值瞬时回填”
- **当前现状**：当把一个新节点（比如 Flashcard）从左侧拖进画布时，这个节点身上的 `data.config` 是空的。直到用户在节点上右键 -> 点击【节点设置】-> 才会把 Manifest 中定义的 default schema 注入并保存进去。
- **还差什么**：
  - 前端画图引擎（Store）在实例化初始节点的瞬间，应当主动去 Manifest 列表中提取所有配置的 `default` 值，并直接挂在这个节点的 `config` 属性下。这样哪怕用户直接点“执行”，也能以默认参数安全跑通。

### 2. 节点快捷控件与侧边栏抽屉的状态同步
- **当前现状**：像 NodeConfigDrawer 是一个非常厚重和完备的“高级设置”。
- **还差什么**：
  - 在节点本身的卡片（Node Card UI）上，暴露出 1~2 个最高频的表单字段快捷调整（比如直接在节点上切换模型，或者输入关键 Prompt），目前没有在两者之间构建优雅的模型双向绑定机制。

## 四、 全仓级技术债务（质量控制生命线）

### 1. 前端“巨石组件”的切片剥离（Blocker 级）
- **`WorkflowCanvas.tsx` (超 800 行)**
  - 承载职责过重（包含了 React Flow 的拖拽校验、连线渲染、React 状态管理、菜单遮罩层等十几个生命周期）。
  - **必做操作**：按纯逻辑钩子拆出 `useNodeDnD.ts`, `useEdgeValidation.ts`, 把各种悬浮工具栏和缩放控制组件做纯粹剥离。
- **`use-workflow-store.ts`**
  - **必做操作**：它是一个巨大的 Zustand 仓库，需强行按 Slices 架构拆分出 `nodesSlice`, `edgesSlice`, `executionTraceSlice`。

### 2. 前端历史 Lint 与 Strict Mode 报警
- **目前堵塞脚本**：`npm run lint` 和 `npm run lint:lines:strict` 会报警，无法成为 CI 里的阻断关卡。
- **待清理罪状**：
  - `react-hooks/exhaustive-deps` 满天飞（旧有代码缺少依赖注入认知）。
  - 违背纯函数原则的副作用，例如在 Render 里直接用 `Date.now()` 或不安全的 ref 改写。
  - 诸多旧业务切片代码还残存未分类类型或 `any` 声明。

### 3. 缺乏后端端到端（E2E）自动化防线
- **当前现状**：只能在开发过程中自己人肉点击“执行工作流”，后端也没有 `pytest` 安装。
- **必做操作**：
  - 核心执行类 `executor.py` 没有任何自动化测试覆盖，它负责各种 JSON 穿梭和分支拦截，这在以后引入复杂控制流（并发 / Timeout 并行处理）时极为危险。安装并配置 Pytest，给出至少 5 套 Mock DAG 节点流回归保证。

---

## 五、 建议的突围序列与组装路径

为防止把这四大深水区熬成大锅乱炖，建议进行**三个明确且有边界的 Sprint (阶段开发批次)**：

### Sprint A: 攻克“巨石”与规范 (纯重构周)
*   **不添加任何新功能**，这是最底线的原则。
*   利用 AI 直接做 `WorkflowCanvas.tsx` 和 `use-workflow-store.ts` 的外科手术式肢解拆分。
*   强攻修改旧文件把 ESLint 债务清完，使得仓库能在本地拥有跑绿的全局检测命令。

### Sprint B: 执行引擎补全与测试网 (后端破局)
*   本地起 Docker 或者在系统中补齐 `pytest`。
*   编写假节点图（Mock DAG）开始回归测试引擎运转。
*   补齐 `loop_map` 缺失多年的阵列下发控制逻辑 (`Scatter` 操作)。

### Sprint C: 打造明星节点，收拢产品价值 (体验攻坚)
*   主攻产品感知层。为 Flashcard 接入 Anki 生态。
*   为被广泛呼吁的 Knowledge Base 实现基于元信息的引用溯源 UI（这在交互重绘上的成本可能很高，但产品核心价值最高）。
*   加上工作流拖拽默认值实例化，真正收口到用户零等待、零干涉体验。
