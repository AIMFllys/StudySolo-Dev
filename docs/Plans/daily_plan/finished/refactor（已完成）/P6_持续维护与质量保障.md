## 一、代码质量维护

### 1.1 TypeScript 零错误策略

**规则**: `tsc --noEmit` 必须零错误。每次 PR/提交前验证。

**措施**:
- [ ] 在 `package.json` 中添加 `"typecheck": "tsc --noEmit"` script
- [ ] 在 CI/CD 中加入 TypeScript 编译检查
- [ ] 定期审查 `@ts-ignore` 和 `as any` 使用

### 1.2 前后端类型同步

**规则**: 后端每新增/修改节点类型，前端必须同步更新。

**机制**:
```
后端 NODE_REGISTRY 变化
  → GET /api/nodes/manifest 返回更新
  → 前端 NodeType 类型 + NODE_TYPE_META + nodeTypes 字典同步
  → 验证: 对比脚本检查两端一致性
```

**推荐脚本** (放在 `scripts/` 下):
```bash
# 对比前后端节点类型
# 后端: 调用 /api/nodes/manifest → 提取 type 列表
# 前端: 解析 workflow.ts 的 NodeType → 提取类型列表
# 比对: 报告差异
```

### 1.3 死代码清理

**定期巡检** (每月一次):
- 搜索 `@deprecated` 标注 → 确认是否可删除
- 搜索未使用的 export → 验证是否有消费方
- 运行 `npx knip` 或 `npx ts-prune` 检查未使用代码

```

  ↓
P4 右侧面板增强 (12-16h)
  ↓
P5 AI 流式生成 (25-32h)
  ↓
P6 持续维护 (ongoing)
```

**总计开发工时: ~80-100h** (不含 P1 UI 重构和 P6 持续维护)
