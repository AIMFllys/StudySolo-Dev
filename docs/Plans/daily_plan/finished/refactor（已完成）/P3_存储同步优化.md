## 一、现状分析

### 1.1 当前同步架构

```
用户操作 (拖拽/连线/配置)
  → isDirty = true
  → 500ms debounce → saveLocal (IndexedDB via localforage)
  → 4000ms debounce → saveCloud (PUT /api/workflow/:id)
  → 成功: isDirty = false, syncStatus = 'synced'
  → 失败: syncStatus = 'error' | 'offline'
```

**文件**: `use-workflow-sync.ts` (232 行)

### 1.2 已知问题

| # | 问题 | 严重度 | 详情 |
|---|------|--------|------|
| S1 | **闭包过期** | 🔴 | `saveLocal` / `saveCloud` 通过 `useCallback` 依赖 `[nodes, edges]`，但 debounce timer 可能执行旧闭包。`pendingSnapshotRef` 模式只在 unmount 路径中使用，save 路径未统一 |
| S2 | **无 Undo/Redo** | 🟡 | P2 新增了 addNode/deleteNode 后用户更需要撤销 |

| S5 | **Online→Offline→Online 回退** | 🟠 | 离线期间多次修改后上线，只 save 最后一份，中间修改丢失 |

## 二、Task 清单

### Phase 3A: 消除闭包陷阱 (2-3h)

#### T3A.1: 统一使用 Ref 快照模式

**问题核心**: `useCallback` 的依赖数组导致每次 nodes/edges 变化都重建回调，旧的 debounce timer 可能引用旧闭包。

**方案**: 将 `saveLocal` 和 `saveCloud` 改为读取 `pendingSnapshotRef.current`（已有但未统一使用）。

```typescript
// 修改前 (闭包陷阱):
const saveLocal = useCallback(async () => {
  const cache = { nodes, edges, ... };  // ← 闭包中的 nodes/edges 可能过期
}, [currentWorkflowId, nodes, edges]);

// 修改后 (统一 Ref):
const saveLocal = useCallback(async () => {
  const { nodes, edges, currentWorkflowId } = pendingSnapshotRef.current;
  if (!currentWorkflowId) return;
  const cache = { nodes, edges, ... };  // ← 始终最新
}, []);  // ← 无依赖，永远不重建
```

**影响范围**: `saveLocal`, `saveCloud` 两个函数。

#### T3A.2: 修复 debounce 生命周期

当前 debounce 在 `useEffect` 中创建，每次 `isDirty` + `nodes` + `edges` + `saveLocal` + `saveCloud` 变化都会清除并重建 timer。统一成 Ref 模式后，effect 依赖简化为 `[isDirty, currentWorkflowId]`。

```typescript
useEffect(() => {
  if (!isDirty || !currentWorkflowId) return;
  
  if (localTimerRef.current) clearTimeout(localTimerRef.current);
  localTimerRef.current = setTimeout(saveLocal, LOCAL_DEBOUNCE_MS);
  
  if (cloudTimerRef.current) clearTimeout(cloudTimerRef.current);
  cloudTimerRef.current = setTimeout(saveCloud, CLOUD_DEBOUNCE_MS);
  
  return () => {
    if (localTimerRef.current) clearTimeout(localTimerRef.current);
    if (cloudTimerRef.current) clearTimeout(cloudTimerRef.current);
  };
}, [isDirty, currentWorkflowId, saveLocal, saveCloud]);
// 由于 saveLocal/saveCloud 不再重建，effect 不会频繁重跑
```

### Phase 3B: Undo/Redo 历史管理 (3-4h)

#### T3B.1: 设计 History Stack

**方案**: 使用 Zustand middleware + 自定义 History

```typescript
// 方案 A: 手动维护 history stack (推荐，更可控)
interface HistoryEntry {
  nodes: Node[];
  edges: Edge[];
  timestamp: number;
  action: string;  // 'add_node' | 'delete_node' | 'move_node' | 'connect' | ...
}

interface HistoryState {
  past: HistoryEntry[];
  future: HistoryEntry[];
  pushHistory: (action: string) => void;
  undo: () => void;
  redo: () => void;
  canUndo: boolean;
  canRedo: boolean;
}
```

#### T3B.2: 集成到 Store

每次 addNode / deleteNode / onConnect / 拖拽位置变化 时，调用 `pushHistory(action)` 保存快照。

**注意**:
- History 最大保留 **50 步** (防止内存膨胀)
- 节点执行产生的 status/output 变化**不计入** History (执行结果不可撤销)
- 位置拖拽使用 **debounce**，连续拖拽只记一次 History

#### T3B.3: 快捷键绑定

| 快捷键 | 操作 |
|--------|------|
| `Ctrl+Z` | `store.undo()` |
| `Ctrl+Shift+Z` / `Ctrl+Y` | `store.redo()` |

### Phase 3C: Crash Recovery UI (2-3h)

#### T3C.1: 监听 crash-recovery 事件

**现状**: `use-workflow-sync.ts` 在检测到 `dirty && local > cloud` 时 dispatch `workflow:crash-recovery` CustomEvent。但**没有任何 UI 组件监听这个事件**。

**方案**: 新建 `CrashRecoveryBanner` 组件

```typescript
// 在 WorkflowCanvasLoader 中监听
useEffect(() => {
  const handler = (e: CustomEvent) => {
    setRecoveryData(e.detail);
    setShowRecoveryBanner(true);
  };
  window.addEventListener('workflow:crash-recovery', handler);
  return () => window.removeEventListener('workflow:crash-recovery', handler);
}, []);
```

**UI**:

### Phase 3E: 保存策略优化 (1-2h)

#### T3E.1: 增量保存 (可选优化)

**当前**: 每次 save 都发送完整的 `{ nodes_json, edges_json }`。

**优化方向**: 对于大型工作流 (>20 节点)，可考虑:
- 前端计算 diff（变化了哪些节点）
- 只发送变化部分

**决策**: 当前阶段暂不实施增量保存。完整保存对于 <50 节点的工作流性能足够。标记为 Stage 3 优化项。

#### T3E.2: 保存失败重试

```typescript
// 添加指数退避重试
async function saveCloudWithRetry(maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      await saveCloud();
      return;
    } catch {
      if (i < maxRetries - 1) {
        await delay(1000 * Math.pow(2, i));
      }
    }
  }
  setSyncStatus('error');
}
```
