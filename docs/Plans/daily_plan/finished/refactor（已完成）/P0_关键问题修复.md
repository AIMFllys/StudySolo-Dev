## 四、存储/同步逻辑隐患排查

### 4.1 use-workflow-sync 闭包陷阱

**现状**: `use-workflow-sync.ts` 中 `saveLocal` 和 `saveCloud` 回调存在潜在的闭包过期问题：

```typescript
// L93: saveLocal 依赖 nodes, edges
const saveLocal = useCallback(async () => {
  // 使用闭包中的 nodes, edges
}, [currentWorkflowId, nodes, edges]);  // ← 每次变化都重建
```

当 `isDirty` 触发 debounce timer 时，如果 nodes/edges 在 timer 期间再次变化，timer 仍会执行旧的 saveLocal（旧闭包），但 React 会因依赖变化重新生成新的 saveLocal，这可能导致：
- debounce timer 调用旧闭包中的 stale 数据
- 云同步可能覆盖最新数据

**修复方案**: 使用 `pendingSnapshotRef` 的模式（已在 unmount 路径中使用）统一到所有保存路径。

### 4.2 Cloud 保存不含节点配置

**现状**: 云保存发送 `{ nodes_json, edges_json }`，但未验证节点的 `data.config` 是否被正确序列化和恢复。

**操作**: 手动测试保存→刷新→加载是否保留所有 node data。
