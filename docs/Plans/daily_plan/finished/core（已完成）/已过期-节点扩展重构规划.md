## 二、重构目标与核心原则

### 2.1 目标

1. **扩展节点池**：从 9 个 P0 节点 → 15+ 个可用节点
2. **补齐学习闭环**：加入"测评"能力（quiz_gen），形成「学 → 练 → 测」闭环
3. **丰富输出形态**：对比表格、思维导图、文件导出等多种知识载体
4. **引入外部数据**：知识库检索 + 联网搜索，让学习不再"凭空臆造"
5. **升级 ai_planner**：让 AI 规划器能调度新节点，自动编排更丰富的工作流

### 2.2 核心原则

| 原则 | 说明 |
|------|------|
| **不改引擎只加节点** | 第一阶段只往 `nodes/` 目录加文件，executor.py 零改动 |
| **一个 PR 只加一个节点** | 保证每次合并都是可测试、可回滚的最小单元 |
| **新渲染器独立文件** | 每个需要新交互的节点，前端单独建一个 Renderer 文件 |
| **向下兼容** | 现有 9 个节点的行为不能有任何变化 |

### 2.3 节点两大类型最终定义

经过深度分析，节点按"内部复杂度"分为两大类：

**专一型节点**：
```
一个 System Prompt + 一个 Mixin 组合 = 完事
特点：改个 prompt.md 就能出一个新节点
代表：outline_gen, summary, content_extract, merge_polish, compare
```

**封装功能型节点**：
```
一个节点对外 = 画布上一个方块
一个节点对内 = 多种子功能的集合体
特点：需要专属的后端 Service + 前端交互组件 + 可能需要新建 DB 表
代表：flashcard, quiz_gen, knowledge_base, export_file
```

### 5.5 `knowledge_base` — 知识库检索（封装功能型，最重）

**核心定位**：让用户用自己的材料学习

> ⚠️ 这不是一个简单的节点，而是一个完整的**子系统**。

**两个独立流程**：

```
流程 A：文件入库（独立管理功能，不在工作流中）

流程 B：在工作流中检索（作为节点参与执行）
  [trigger_input] → [knowledge_base] → [content_extract]
                          |
                    不调 LLM，调 retriever service
                    输出相关片段 + 引用来源
```

**后端新增文件**：
```
services/
├── file_parser.py          ← pypdf + python-docx
├── text_chunker.py         ← 按标题分块 + 512 token overlap
├── embedding_service.py    ← text-embedding-v4 接入
└── knowledge_retriever.py  ← 三层漏斗检索

api/knowledge.py            ← POST upload / GET query / GET list / DELETE

nodes/input/knowledge_base/
├── __init__.py
├── node.py                 ← KnowledgeBaseNode（不走 LLM）
└── prompt.md               ← 说明文档
```

**数据库新增表（Supabase Migration）**：
```sql
-- kb_documents：文档元信息
-- kb_document_summaries：AI 摘要 + 核心观点 + 目录
-- kb_document_chunks：分块后的文本
-- kb_chunk_embeddings：分块向量（pgvector）
-- kb_summary_embeddings：摘要向量
```

**预计工时**：2-3 周

## 六、引擎层重构

### 6.1 第一阶段（P1 节点期间）：最小改动

**executor.py 无需大改**，但需要验证以下场景：

```python
# 当 node 的 execute() 不使用 llm_caller 时：
async for token in node_instance.execute(node_input, call_llm):
    # knowledge_base/web_search/export_file 的 execute() 
    # 内部直接调 service，不碰 call_llm
    # 但 executor 照常收集 token 并推 SSE
    full_output += token
    yield sse_event("node_token", {"node_id": node_id, "token": token})

# 只要这些节点的 execute() 是 AsyncIterator[str]
# 并且 yield 出文本，引擎就不需要改
```

**可能的小改**：
- `config.yaml` 需要为非 LLM 节点添加路由条目（即使不用模型）
- 或者在 executor.py 中判断 `is_llm_node` 属性决定是否走路由

### 6.2 第二阶段（P2 引擎增强）：大改

| 改动 | 内容 |
|------|------|
| 条件分支执行 | executor 识别 `logic_switch` 节点后，根据输出仅执行某一路分支的下游 |
| 循环迭代 | executor 识别 `loop_map` 节点后，对数组逐项执行子 DAG |
| 超时控制 | 单节点超时机制 + 全链路超时 |
| 并行执行 | 同层无依赖节点并行执行（当前是逐个串行） |

### 6.3 NodeInput 扩展（可选优化）

```python
# 当前
@dataclass
class NodeInput:
    user_content: str              # node_data["label"]
    upstream_outputs: dict[str, str]
    implicit_context: dict | None

# 建议扩展
@dataclass
class NodeInput:
    user_content: str
    upstream_outputs: dict[str, str]
    implicit_context: dict | None
    node_config: dict | None = None   # ← 新增：节点配置参数
    # 用于 quiz_gen 传递 { "types": ["choice"], "count": 10, "difficulty": "medium" }
    # 用于 export_file 传递 { "format": "pdf" }
```

## 八、ai_planner 提示词升级方案

### 8.1 当前 ai_planner 可用节点池

```
outline_gen, content_extract, summary, flashcard, chat_response, write_db
```

### 8.2 升级后可用节点池

```
节点类型定义（ai_planner 可选用的节点）：
- outline_gen:     📋 生成结构化学习大纲
- content_extract: 📖 深度提炼某一章节的知识点
- summary:         📝 多源归纳总结（汇聚多个上游输出）
- flashcard:       🃏 生成记忆闪卡（Q&A 对）
- quiz_gen:        📝 生成测验题目（选择题/判断题/填空题混合）
- compare:         ⚖️ 多概念多维度对比辨析表格
- mind_map:        🧠 知识结构可视化思维导图
- merge_polish:    ✏️ 多段文本润色合并为连贯长文
- chat_response:   💬 自由对话回答
- write_db:        💾 保存结果到数据库
```

### 8.3 ai_planner prompt 新增规则

```markdown
新增编排规则：
1. 当用户需求涉及"对比"、"区别"、"异同"时，使用 compare 节点
2. 当用户需求涉及"测验"、"考试"、"练习"时，在最后加 quiz_gen 节点
3. 当有 ≥3 个并行 content_extract 时，在下游加 merge_polish 汇聚
4. 当用户需求涉及"可视化"、"导图"、"结构化"时，使用 mind_map 节点
5. quiz_gen 的 label 中要包含题型和数量指令（如："📝 生成 10 道选择题和判断题"）
```

## 十一、风险与应对策略

| 风险 | 影响 | 应对 |
|------|------|------|
| 非 LLM 节点的 execute() 可能和引擎冲突 | knowledge_base/web_search/export_file | 先写单测模拟，确认引擎兼容再集成 |
| quiz_gen JSON 输出不稳定 | 题目结构残缺 | 走格式严格链（route_chain A）+ JsonOutputMixin 修复 |
| ai_planner 不会用新节点 | 规划出的工作流还是只用老节点 | 在 prompt 中添加明确的"什么时候用哪个节点"规则 |
| knowledge_base 向量化成本 | embedding API 调用费用 | 使用国内 text-embedding-v4（阿里云），价格为百万 token ¥0.7 |
| 前端渲染器交互复杂度 | QuizRenderer 需要大量状态管理 | 使用 useReducer 管理答题状态机 |
| config.yaml route 膨胀 | 节点越多配置越长 | 保持现有结构不变，只追加条目 |
