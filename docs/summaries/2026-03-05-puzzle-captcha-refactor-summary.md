# 滑动拼图验证码重构与代理链路修复总结

> 📅 日期：2026-03-05  
> 📌 关联版本：v0.3.3

## 背景与目标
项目中原有的 `SliderCaptcha` 存在大面积的功能与设计漏洞：坐标因为 CSS 伸缩发生了无法弥补的时空偏移，导致在验证时无论是何种位置，计算比对结果永远相距甚远。
同时伴随着前端 Next.js 请求转发后端时常发性的假报错。
核心重构目标是**彻底落地可商用级别、高精准性、视觉高级（含动态拼图骨架与主题）的拼图组件，并打通被遮掩很久的前端代理 500 连接异常**。

## 实现过程与架构
1. **纯粹的双轨并行 Canvas 体系**：摒弃了之前由于使用一单独主图+绝对位移叠加的小 DOM 方块导致的时空域重叠与漂移。改为：一号画布专门铺垫渲染带有阴影半透缺口的背景色、二号画布叠在上方单独渲染利用 `clip`, `getImageData` 提纯出像素级一比一的裁版拼片。比对由 CSS Right 直接锚定。
2. **前后端代理层追本溯源**：发现即便利用最精确的前端手段强行校直位置（让 `spliced` 状态成功进入 fetch），仍然会因为网络服务 500 （非前端 JS 报错）直接走到 `catch` 显示 “拼合不准确”。随后层层剥离出 `next.config.ts` 在 Node.js 框架下发酵的环境兼容故障。

## 核心重构梳理
- **Canvas 精细曲线绘制参数更正**：从之前的单纯近圆切口 `0.3 * PI` 重新代入了参考级严密的切向角度值（诸如 `Math.PI * 0.72` 至 `2.26 * PI` 等多处精确拟合），生成真实、带凹凸锁扣结构的拼图样貌。
- **Seed PRNG 一致性与防重放机制**：前端通过组件级 `mulberry32` 防抖加随机种，兼具展示无断层且每次拉锯和刷新都提供完全独立形状。解决因为渲染依赖 Server-Side 引发的 React Hydration 报错（改用 `mounted` 包裹隔绝 SSR 执行）。
- **`next.config.ts` 下的 `127.0.0.1` 锚固行动**：将原先回退时指向 `http://localhost:2038` 的所有转发链路全部显式更改为 `127.0.0.1:2038`，杜绝任何 Node.js 版本（比如 v17+）将其解析成 `::1` 进而被只负责监听 IPv4 (0.0.0.0) 的 uvicorn 拒绝（引发致命的底层代理 ECONNREFUSED），重铸打通数据大动脉通道。
- **FastAPI 表单验证层补丁**：定位到后端 `auth/send-code` 与鉴权组件等由于 FastAPI Form 处理必须要借助 `python-multipart` 在虚拟环境安装却常常忽视的事实，直接沉淀落实进 `requirements.txt`。

## 踩坑记录
1. **被前端错误提示掩盖的网络阻断**：当网络或者代理因为 HTTP 500 或者 400 不通畅的时候，我们早期的容错非常简陋地把所有的 `try-catch` fallback 定位在“拼力不足”，导致开发者也跟着误入歧途在几何角度与算法算子上死扣。我们新版本不仅修复了画布本身确实不对准的核心顽疾，更将网络不通（`apiError`）区分，显示"验证服务异常，请稍后重试"。
2. **MIME Type 之祸**：Google Fonts 的链接 `https://fonts.googleapis.com/...` 其实吐出来的是 `text/css`。此前却通过 `next/script` 引用且由于严格的 Mime 类型验证发生拒载，引发后台的报毒。修改为了标配的 `<link rel="stylesheet">`。

## 经验总结
- **问题排查要警惕“文不对题”**：如果一个界面行为提示是“前端坐标不准计算错”，但在 DevTools 的抓包和 Console 层可能是 HTTP 500 ，应当始终优先查看服务端和代理转发，不被表象蒙蔽去钻研前端。
- **Node.js 本地开发中的 IPv6 问题变数**：Node.js 17+ 环境下配置 `proxy` / `rewrites` 请毫不犹豫地使用直接对应的 IP 地址 (`127.0.0.1`) 替代 `localhost`。特别是涉及多语言微服务联动（如 Python FastAPI，默认 `0.0.0.0`），这能省去巨量的代理转发抓虫心智负担。
